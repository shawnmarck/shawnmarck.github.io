Strategy Update — April 2026

Maximizing a Prepaid ZAI Plan Without Queuing Everything

Running a ZAI GLM Coding Plan (Max tier) is excellent value — you get GLM-5.1, GLM-5-Turbo, and the wider 4.7 family for a flat monthly rate. The catch: every GLM model is capped at 1 concurrent request. In a multi-agent setup where explore, atlas, librarian, and sisyphus-junior can all fire at once, that single-slot limit creates invisible queues and 429 timeouts that silently kill throughput.

The fix is a two-layer strategy. First, push latency-sensitive but less-reasoning-heavy agents (document-writer, frontend-ui-ux-engineer) onto Google Gemini 3 Flash — it supports 10 concurrent requests and is faster for prose and UI work anyway. Second, enable runtime fallback so that when a ZAI model is occupied, agents automatically retry on the next model in their fallback chain — Gemini 3 Flash, then Kimi — without surfacing errors to the user.

High-reasoning agents (sisyphus, prometheus, metis) stay on GLM-5.1 max because the deep-thinking variant is where the prepaid plan earns its keep. They run sequentially anyway — you don't plan and execute in parallel — so the concurrency limit is rarely a problem in practice. Execution agents (atlas, explore, librarian, sisyphus-junior) move to GLM-5-Turbo, which is faster and cheaper on the prepaid plan, with Gemini as the high-volume escape valve when turbo slots fill.

A new turbo-exec category joins quick for high-frequency parallel sub-task dispatch, both routing to GLM-5-Turbo with Gemini fallback. The writing category switches to Gemini 3 Flash as primary — the original GLM-4.7-Flash was noticeably slow for document generation.

This configuration is managed via the opencode-omo-plugin — an OpenCode plugin that ships the /omo command and omo-config skill for interactive config management directly from the terminal.
