What the wat-vm is#

wat-vm is a standalone binary (src/bin/wat-vm.rs, 846 lines). It is the "second heartbeat" — a process that reads candle data from files, feeds it to a grid of parallel observers and brokers, lets them learn, and then collects them back at shutdown. The CLI summary says it plainly: "Candles in, observers learn, results discarded." It is a training/learning run, not a live trading system.

CLI interface#

wat-vm --stream SOURCE:TARGET:PATH [--max-candles N] [--dims D] [--recalib-interval R] [--swap-fee F]
  • --stream: repeatable. Format SOURCE:TARGET:PATH (e.g., binance:BTCUSDT:data/btc.csv). Multiple streams are processed sequentially.
  • --max-candles: per-stream cap. Each pipeline owns its limit.
  • --dims: vector dimension (default 10000).
  • --recalib-interval: observations between recalibrations (default 500).
  • --swap-fee: venue fee fraction (default 0.0010 = 10 bps).

Signal handling#

A single static AtomicBool named STOP. A C signal handler (via libc::signal) sets it on SIGINT/SIGTERM. The main loop reads it each iteration. Nothing else touches it.

Startup sequence (main)#

  1. Parse args. Compute num_market = config::MARKET_LENSES.len(), num_position = config::POSITION_LENSES.len(), noise_floor = 5.0 / sqrt(dims), num_brokers = num_market * num_position.
  2. Console. Create num_console = streams + num_market + num_position + num_brokers + 1 console handles. Claim the main handle and stream handles first.
  3. Database. Create a SQLite file at runs/wat-vm_<timestamp>.db. Create num_market + num_position + num_brokers + 1 unbounded queues, merge into a mailbox receiver, spawn the database driver (batch size 100, rate-gated telemetry every 5 seconds).
  4. Encoding cache. Create a VectorManager (dims) and ScalarEncoder (dims). Spawn a cache<ThoughtAST, Vector> LRU cache (256K capacity, rate-gated telemetry via a db sender).
  5. Reserve handle pools. Pre-allocate console/cache/db handles for position observers and brokers so that market observer wiring gets whatever remains.
  6. Learn queues. Create N×M unbounded queues for market learn (ObsLearn) and M×N for position learn (PositionLearn). Merge into mailbox receivers per observer. Create N×M trade-state queues (TradeUpdate), merge into mailbox receivers per position observer.
  7. Wire market observers. wire_market_observers() — for each of N market observers: create a bounded candle input queue (capacity 1), M bounded MarketChain fan-out queues, a topic that fans the observer's output to all M position queues. Spawn a thread running market_observer_program. Assert all pooled handles were claimed.
  8. Wire position observers. wire_position_observers() — transpose the [mi][ei] receiver grid into [ei][mi]. For each of M position observers: build N PositionSlots, each with input from one market observer's topic queue and an unbounded output queue. Spawn a thread running position_observer_program.
  9. Wire brokers. wire_brokers() — flatten the 2D learn-tx grids. For each of N×M brokers: assign the output_rx from position observer, the market learn tx, position learn tx, and trade tx. Spawn a thread running broker_program.
  10. Build pipelines. For each --stream: open a CandleStream, create an IndicatorBank.
  11. Run the main loop (see below).
  12. Shutdown (see below).

The main loop#

Per-stream, sequential:

while count < max && !STOP:
    candle = pipeline.stream.next()  // raw OHLCV from file
    candle = pipeline.bank.tick(&ohlcv)  // enrich with indicators
    for each market observer candle_tx:
        send ObsInput { candle, window, encode_count }
    every 100 candles: log throughput

Key details:

  • The main thread reads candles sequentially and fire-and-forget sends to all N market observers. It does not wait for results. This is a push architecture — the main thread is the sole candle producer.
  • ObsInput carries: candle (enriched), window (Arc<Vec<Candle>> — currently always length 1), encode_count (global monotonic counter).
  • The IndicatorBank::tick() enriches raw OHLCV into a full Candle struct.

Data flow through the grid#

flowchart TD
    main["Main thread\nsend ObsInput"] --> mkt_obs["Market Observer i\n(thread)\nencode -> predict"]
    mkt_obs -->|"Topic fan-out\nsend MarketChain"| pos0["Position Observer 0"]
    mkt_obs --> pos1["Position Observer 1"]
    mkt_obs --> posM["Position Observer M"]
    pos0 -->|"send MarketPositionChain"| broker["Broker [i, ei]\n(thread)"]
    pos1 --> broker
    posM --> broker
    broker -.->|"ObsLearn -> market observer i's mailbox"| mkt_obs
    broker -.->|"PositionLearn -> position observer ei's mailbox"| posM
    broker -.->|"TradeUpdate -> position observer ei's trade mailbox"| posM
  • MarketChain flows from market observers to position observers via topics (fan-out).
  • MarketPositionChain flows from position observers to brokers via unbounded queues.
  • Learn signals flow back from brokers to observers via unbounded queues merged into mailboxes (fan-in).
  • Trade updates flow from brokers to position observers via unbounded queues merged into mailboxes.

Shutdown sequence#

Ordered teardown:

  1. Drop candle_txs — market observers see disconnect on their input queues.
  2. Join market observer threads — collect trained observers back, print experience/resolved counts.
  3. Drop topic_handles — topics drain remaining messages to position observers.
  4. Join position observer threads — collect trained observers back, print trail/stop experience.
  5. Broker input queues disconnect automatically when position observers drop their output senders. Join broker threads — collect trained brokers back, print trade counts, grace rate, expected value, open papers.
  6. Print db path.
  7. Print cache summary (hits/misses/rate).
  8. Drop stream handles. Join cache driver (must join before db because its emit closure holds a db_tx). Join db driver. Drop main handle. Join console driver.

Chain types (src/programs/chain.rs, 40 lines)#

Two structs. Pure data. No methods. No behavior. The module doc says: "Each stage produces a new value. The type IS the proof of which stage produced it. No methods. No behavior. Values on the wire."

MarketChain (Clone)#

What the market observer produces. What the position observer receives.

Field Type Meaning
candle Candle The enriched candle that triggered this step
window Arc<Vec<Candle>> Window of recent candles (currently length 1)
encode_count usize Global monotonic encode counter
market_raw Vector Raw encoding of market features
market_anomaly Vector Anomaly component (deviation from noise subspace)
market_ast ThoughtAST The abstract syntax tree that was encoded
prediction holon::memory::Prediction Direction prediction (Up/Down)
edge f64 Edge/confidence score

MarketPositionChain#

What the position observer produces. What the broker receives. Contains all MarketChain fields (flattened, not composed) plus position-specific fields:

Field Type Meaning
candle Candle Same candle
window Arc<Vec<Candle>> Same window
encode_count usize Same counter
market_raw Vector Same market raw encoding
market_anomaly Vector Same market anomaly
market_ast ThoughtAST Same market AST
market_prediction holon::memory::Prediction Same prediction (renamed from prediction)
market_edge f64 Same edge (renamed from edge)
position_raw Vector Position observer's raw encoding
position_anomaly Vector Position observer's anomaly
position_ast ThoughtAST Position observer's AST
position_distances Distances Trailing stop and safety stop distances

Note: MarketPositionChain does not implement Clone. The market fields are duplicated (not borrowed), making this struct self-contained but large — a deliberate "values on the wire" design with no shared references.

Thread count#

For one stream with N market observers, M position observers: N + M + (N×M) brokers + database + cache + console = N + M + N×M + 3 threads minimum. With default 6 market lenses and 2 position lenses: 6 + 2 + 12 + 3 = 23 threads. The comment in CLAUDE.md says "30+ threads" — the additional threads likely come from multiple streams or internal service threads.

Messaging primitives#

Three types, zero Mutex:

  • queue_bounded / queue_unbounded — point-to-point channels (mpsc via crossbeam-style implementation).
  • topic — one publisher, multiple subscribers. Fan-out. The topic handle must live until shutdown.
  • mailbox — merges multiple receivers into one. Fan-in. Used for learn queues (multiple brokers → one observer).

Resources allocated from pools#

Console, cache, and database handles are pre-allocated into HandlePools. Each program thread claims exactly one of each. Pools assert finish() — all handles must be claimed or the program panics (orphan detection to prevent deadlocks).
