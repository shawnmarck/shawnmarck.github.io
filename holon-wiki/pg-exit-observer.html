Position Observer — Pure Lens Carrier EXPLAIN

TL;DR: A PositionObserver is a pure lens carrier — one field (lens: PositionLens), one constructor, zero behavior. It does not learn, does not predict, does not encode. The domain struct is 17 production lines. The real work happens in the program thread (position_observer_program.rs, 213 lines), which composes position-specific vocabulary with market thoughts and sends raw ThoughtAST facts downstream. The broker is the accountability unit.

"Does not learn. Does not predict. Quality is measured through the BROKER's curve." — module docstring

Anatomy#

The domain struct is deliberately minimal. Everything that was stripped — reckoners, noise subspace, incremental cache, learning — is gone.

Field Type Role
lens PositionLens Which judgment vocabulary: Core (10 vocab facts) or Full (13 vocab facts)

That's it. One field.

pub struct PositionObserver {
    pub lens: PositionLens,
}

impl PositionObserver {
    pub fn new(lens: PositionLens) -> Self {
        Self { lens }
    }
}

What Was Removed#

The old PositionObserver had 6 fields, 291 lines, two continuous Reckoners, a noise subspace, an incremental bundle cache, prediction methods (reckoner_distances), and learning methods (observe_distances). All of this was reaped. The rationale:

  • Distances never worked. The continuous reckoners predicted trail/stop distances that never produced useful edge.
  • Learning without signal is noise. The position observer received no direct feedback — only indirect signals through the broker.
  • The broker IS the accountability unit. It owns paper trades, scalar accumulators, and the three-tier cascade. The observer should not duplicate that.

The Program Thread#

The program (position_observer_program.rs, 213 lines) is where the position observer actually does work. It runs in its own thread, processing one slot per broker connection per candle round.

PositionSlot#

Field Type Role
input_rx QueueReceiver<MarketChain> Receives from one market observer (via topic fan-out)
output_tx QueueSender<MarketPositionChain> Sends to exactly one broker (point-to-point queue)

Per-Candle Loop#

flowchart TD
    recv["recv MarketChain from slot"] --> facts["position_lens_facts(lens, candle, scales)"]
    facts --> clone["clone base_facts for each slot"]
    clone --> extract["collect_facts from market_ast → encode → cosine"]
    extract --> anomaly["Cosine vs market_anomaly — bind matching facts as (market ...)"]
    anomaly --> raw["Cosine vs market_raw — bind matching facts as (market-raw ...)"]
    raw --> send["Compose MarketPositionChain → send to broker"]
  1. Receive — slot.input_rx.recv() — gets a MarketChain from the upstream market observer.
  2. Position facts — position_lens_facts(&lens, &candle, &mut scales) — computes lens-specific vocabulary. Computed once per candle (first slot), then cloned for subsequent slots.
  3. Market fact extraction — Collects facts from chain.market_ast, encodes each into a vector, then does two cosine passes:
    • Against chain.market_anomaly — facts above noise_floor threshold are bound as (market fact).
    • Against chain.market_raw — facts above noise_floor threshold are bound as (market-raw fact).
  4. Send downstream — Composes a MarketPositionChain with all fields forwarded plus position_facts: Vec<ThoughtAST>. No encoding. The broker encodes when it composes with anxiety.

Key Design Decision: No Encoding Here#

The position observer sends raw ThoughtAST facts downstream. It does not encode them into vectors. Encoding happens in the broker, which composes position facts with market thoughts and anxiety. This keeps the position observer as pure middleware — vocabulary composition only.

The Lens — What the Observer Sees#

Each observer wears a PositionLens that determines its vocabulary (from position_lens_facts in lens.rs):

Lens Vocab Sources Fact Count
Core exit-regime (8) + exit-time (2) 10
Full exit-regime (8) + exit-time (2) + phase-current (1) + phase-series (1) + phase-scalars (1) 13

"Core: lean — regime + time. 5 trade atoms downstream." — T0-005 "Full: rich — regime + time + phase series. 13 trade atoms downstream." — T0-005

The "trade atoms" (5 for Core, 13 for Full) arrive through the broker's trade pipe, not from position_lens_facts. The vocabulary facts above are what the position observer produces — the trade atoms are what the broker consumes.

MarketPositionChain#

Defined in chain.rs. What the position observer produces; what the broker receives.

Field Type Source
candle Candle Forwarded from MarketChain
window Arc<Vec<Candle>> Forwarded from MarketChain
encode_count usize Forwarded from MarketChain
market_raw Vector Forwarded from MarketChain
market_anomaly Vector Forwarded from MarketChain
market_ast ThoughtAST Forwarded from MarketChain
market_prediction Prediction Forwarded from MarketChain.prediction
market_edge f64 Forwarded from MarketChain.edge
position_facts Vec<ThoughtAST> Produced here — lens vocab + market-extracted facts

"No encoding. No vectors. No distances. The broker encodes." — chain.rs docstring

There are no position_raw, position_anomaly, position_ast, or position_distances fields. The position observer enriches with AST facts only.

Position in the Pipeline#

flowchart TD
    market_obs["Market Observer"] -->|"\nTopic (MarketChain)"| pos_prog["position_observer_program()"]
    pos_prog -->|"Queue\n(MarketPositionChain)"| broker["Broker"]
    broker -.->|"Mailbox\n(ObsLearn)"| market_obs
  • Input: MarketChain from each market observer (via Topic, fan-out 1→N).
  • Output: MarketPositionChain to each broker (via Queue, point-to-point).
  • No feedback path — the position observer receives no learn signals. The broker handles all accountability.
