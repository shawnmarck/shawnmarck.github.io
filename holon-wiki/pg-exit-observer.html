Exit Observer — Perceiving When to Leave a Position EXPLAIN

TL;DR: A PositionObserver (also called "exit observer") predicts two continuous values — trailing stop distance and safety stop distance — from a position thought vector using two independent continuous <a class="wikilink" href="pg-reckoner.html">Reckoner</a>s. It learns from hindsight-optimal distances determined by simulation. Each observer wears a <a class="wikilink" href="pg-lens.html">PositionLens</a> that selects its trade-state vocabulary. It has a <a class="wikilink" href="T0-010-subspace.html">noise subspace</a> for anomaly detection despite its own docstring claiming it doesn't. The <a class="wikilink" href="pg-broker.html">broker</a> owns the production cascade (reckoner → accumulator → default); the observer only exposes raw reckoner queries. M per trading system; N×M brokers total.

#Anatomy

Component Type Role
lens <a class="wikilink" href="pg-lens.html">PositionLens</a> Which trade-state vocabulary this observer uses: Core (5 trade atoms) or Full (13 trade atoms)
trail_reckoner <a class="wikilink" href="pg-reckoner.html">Reckoner</a> (continuous) Predicts trailing stop distance. Named "trail-{lens}". 10 buckets.
stop_reckoner <a class="wikilink" href="pg-reckoner.html">Reckoner</a> (continuous) Predicts safety stop distance. Named "stop-{lens}". 10 buckets.
default_distances Distances Fallback {trail: f64, stop: f64} — percentage of price, scale-free. Returned when reckoners have no experience.
noise_subspace <a class="wikilink" href="T0-010-subspace.html">OnlineSubspace</a> 8 principal components. Learns the background distribution for this exit lens. The anomaly is what the subspace cannot explain.
incremental IncrementalBundle Optimization cache — running element-wise sums + last-facts cache for incremental encoding. Not cognition.

"Intentionally simpler than MarketObserver." — module docstring (though the struct does include a noise subspace, contradicting this claim — see Discrepancies).

#Prediction (reckoner_distances)

One public prediction method: reckoner_distances(position_thought: &Vector) -> Option<Distances>.

  1. Check experience — if either reckoner has zero experience, return None.
  2. Query trail — trail_reckoner.query(position_thought) → trailing stop distance.
  3. Query stop — stop_reckoner.query(position_thought) → safety stop distance.
  4. Return — Some(Distances { trail, stop }).

Tier 1 only — raw reckoner output with no fallback. The broker owns the full three-tier cascade:

Tier Source Owner
1 Reckoner query (this method) Position observer
2 Scalar accumulators (per-broker) Broker
3 default_distances Broker

Per <a class="wikilink" href="pg-signal-propagation.html">Proposal 026</a>, queries operate on position_thought only — no composition with market thought at query time.

"The broker owns the full cascade because it owns the scalar accumulators. The reckoner answer comes from outside (the position observer's continuous reckoner), but tiers 2 and 3 are broker-owned." — T0-018

#Learning (observe_distances)

One learning method: observe_distances(position_thought: &Vector, optimal: &Distances) -> (f64, f64, f64, f64, f64, f64).

  1. Pre-learning prediction — query each reckoner for what it would predict given current state. If no experience, use default.
  2. Error computation — |predicted - optimal| / optimal.max(0.0001) for both trail and stop. Normalized error. Read-only diagnostic — no feedback loop.
  3. Observe — trail_reckoner.observe_scalar(position_thought, optimal.trail, 1.0) and stop_reckoner.observe_scalar(position_thought, optimal.stop, 1.0). Weight is always 1.0 — each observation counts once.
  4. Return — (trail_error, stop_error, predicted_trail, predicted_stop, optimal_trail, optimal_stop).

The "optimal" distances come from hindsight — the simulation module sweeps candidate trail/stop values against actual price histories to determine what would have been ideal.

#How the Continuous Reckoner Learns

Each continuous <a class="wikilink" href="pg-reckoner.html">Reckoner</a> uses bucketed nearest-neighbor regression:

  • 10 buckets span the discovered value range. The first observation seeds a tiny range (±0.001); out-of-range values expand it; every 100 observations the range contracts to where alive weight lives ("breathing range").
  • Implicit decay — every observation decays all buckets by 0.999. "Every observation is a tick of time."
  • Query — cosine against bucket prototypes, soft-weight top-3, interpolate. O(K×D), constant in observation count.
  • Range is data-driven — "No magic min/max. The data teaches the range. K is the only parameter." — T0-008

#Noise Stripping

strip_noise(thought: &Vector) -> Vector — converts to f64, runs noise_subspace.anomalous_component(), converts back. Returns what the subspace cannot explain. Same pattern as the <a class="wikilink" href="pg-market-observer.html">market observer</a>.

#The Lens — What the Observer Sees

Each observer wears a <a class="wikilink" href="pg-lens.html">PositionLens</a> that determines its trade-state vocabulary (T0-005):

Lens Trade Atoms Description
Core 5 Excursion, retracement, age, peak-age, signaled. The consensus view.
Full 13 All Core atoms + 3 phase biography atoms. All three voices plus phase context.

"Core: lean — regime + time. 5 trade atoms downstream." — T0-005 "Full: rich — regime + time + phase series. 13 trade atoms downstream." — T0-005

N is the number of market observers (one per <a class="wikilink" href="pg-lens.html">MarketLens</a> variant); M is the number of exit observers (one per <a class="wikilink" href="pg-lens.html">PositionLens</a> variant). N×M brokers bind every market observer to every exit observer.

#Discrepancies

The module docstring (lines 1–5) states: "Intentionally simpler than MarketObserver. No noise-subspace, no curve, no engram gating."

The struct does have a noise_subspace field and a strip_noise method. The docstring contradicts the code. All 7 tests confirm the noise subspace is initialized and functional.

#Comparison — Exit Observer vs. Market Observer

Property Market Observer Exit Observer
Prediction type Discrete (Up/Down) Continuous (trail distance, stop distance)
Reckoner mode Discrete, 2 labels Continuous, 10 buckets each (×2 reckoners)
Noise subspace 8 components 8 components
Conviction curve Yes — self-evaluates accuracy No curve mechanism
Engram gate Yes — gates long-term memory No engram gating
Learning source Market outcomes (actual direction) Hindsight simulation (optimal distances)
Lens <a class="wikilink" href="pg-lens.html">MarketLens</a> (11 variants, 3 schools) <a class="wikilink" href="pg-lens.html">PositionLens</a> (2 variants: Core, Full)

#Position in the Pipeline


The exit observer sits between market observers and brokers in the concurrent pipeline:


flowchart TD
    market_obs["Market Observer"] -->|"Topic\n(MarketChain)"| position_obs["Position Observer"]
    position_obs -->|"Queue\n(MarketPositionChain)"| broker["Broker"]
    broker -.->|"ObsLearn\n(Mailbox)"| market_obs
    broker -.->|"PositionLearn\n(Mailbox)"| position_obs



    

  • Input: <a class="wikilink" href="pg-chain-types.html">MarketChain</a> from each market observer (via <a class="wikilink" href="pg-services.html">Topic</a>, fan-out 1→N).
    • 

  • Output: <a class="wikilink" href="pg-chain-types.html">MarketPositionChain</a> to each broker (via <a class="wikilink" href="pg-services.html">Queue</a>).
    • 

  • Feedback: PositionLearn signals from brokers (via <a class="wikilink" href="pg-services.html">Mailbox</a>, fan-in N→1).
    • 



Each arrow is a <a class="wikilink" href="pg-services.html">Queue</a>. The fan-out uses <a class="wikilink" href="pg-services.html">Topic</a> (1→N). The learn-signal aggregation uses <a class="wikilink" href="pg-services.html">Mailbox</a> (N→1).
