Proposal System EXPLAIN

TL;DR#

The wat language evolves through a formal proposal process. Each proposal describes a problem, proposes alternatives, and is independently reviewed by two designers — Brian Beckman (algebraic authority) and Rich Hickey (simplicity authority). A datamancer arbitrator synthesizes their reviews into a final resolution, sometimes overriding both. Eighteen proposals have been resolved to date (March–April 2026), shaping the language's Structural Forms, core algebra, and concurrency model.

Anatomy of a Proposal#

Every proposal is a PROPOSAL.md with seven required sections:

  1. Scope — classified as core, structural, or following from a prior proposal.
  2. The Current State — what exists today, with code examples from the enterprise codebase or the wat language runtime.
  3. The Problem — a concrete gap, often discovered by automated wards (gaze, forge, sever, scry, reap) or by tracing type violations between wat specs and the Rust implementation.
  4. The Proposed Change — two to three alternatives with code examples.
  5. The Algebraic Question — does this compose with the existing monoid (bind/bundle)? Does it introduce new algebraic structure? Every proposal must address this.
  6. The Simplicity Question — Hickey's test: "Is this simple or easy?" What is being complected? Could existing primitives solve it?
  7. Questions for Designers — specific yes/no and tradeoff questions for the review panel.

The Review Panel#

Brian Beckman — Algebraic Authority#

Frames proposals in category theory: monoids, coalgebras, functors, catamorphisms, universal constructions. His recurring questions: "Does this compose?" and "What categorical construction is this?" He tends to see mathematical structure where Hickey sees convention.

Key positions:

  • Framed fold as the catamorphism (accepted as proposal 002).
  • Framed protocols as "type classes — the third construction alongside struct (product) and enum (coproduct)" (014).
  • Argued journal operations are co-generators of a coalgebra and belong in core (004).

Overridden on: struct field types (007 — wanted mandatory types; untyped won) and defservice (04/003 — datamancer showed functions + select suffice).

Rich Hickey — Simplicity Authority#

Applies "simple made easy" — distinguishes simple (one concept, one job) from easy (familiar, convenient). Protects the six Primitives from growth. His recurring questions: "Does this complect?" and "Could existing forms do this?" He tends to reject new forms when discipline or data restructuring suffices.

Key positions:

  • Rejected defprocessor flatly: "The function shape IS the declaration" (001).
  • Won on untyped struct fields: "Types are documentation, not enforcement" (007).
  • Won keyword-as-function access: "Keywords read like English" (007).

Overridden on: journal operations in stdlib (004 — reversed after opacity argument), nil (013 — datamancer chose structural absence), and channels (003 — deleted as contradicting fold-based design).

The Datamancer — Arbitrator#

Not a formal reviewer but a synthesizing role visible in resolutions. Appears when the two designers disagree or when neither option is right. The datamancer has overruled both designers:

  • Proposal 013 — chose "structural absence" with the ? field convention, rejecting both Clojure truthiness and Rust's nil separation.
  • Proposal 04/003 — showed that select + functions + spawn subsume defprocess and defservice entirely, retiring both forms.

Resolution#

A RESOLUTION.md synthesizes the two reviews into one of four outcomes:

Verdict Meaning
ACCEPTED With specific form syntax, placement, and constraints.
REJECTED With open questions forwarded to future proposals.
ACCEPTED with tension Accepted, but a specific disagreement is deferred.
SUPERSEDED The debate revealed a simpler answer (see 04/003).

Resolution documents record who was overridden and why.

Design Principles (Emergent)#

The proposal system has produced six stable principles:

  1. Transparency to Rust. Form names match Rust (struct, not defstruct). Access syntax maps obviously (:field record → record.field). The wat spec should be readable alongside the implementation.

  2. Protect the algebra. The six Primitives and journal coalgebra are the foundation. Proposal 001 was rejected to prevent a seventh primitive. Only fold was added — as a control form, not a primitive.

  3. Simple > easy. Convenience is rejected if it complects concerns. Variadic update (012) was accepted because it completes an existing form. defprocessor (001) was rejected because it packaged four concerns into one.

  4. Types are documentation, not enforcement. Structs are untyped (007). Protocols check arity, not return types (014). when without else generates Option<T> — the compiler infers; the specifier doesn't annotate (013).

  5. Opacity creates boundaries. If a type is opaque, its interface operations are core (004). This rule determined that observe/predict/decay are core, not stdlib.

  6. The four-layer model (003): Syntax (compiler builtins), Core (algebra), Stdlib (derived operations), Userland (domain code). Each layer has a boundary rule — stdlib cannot import from userland; core can only be extended by proposal.
