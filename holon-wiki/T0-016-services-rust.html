What They Are

Three messaging primitives. Point-to-point (Queue), fan-out (Topic), fan-in (Mailbox). All built on crossbeam::channel. Zero Mutex. Concurrency through message passing alone.

Queue — Point-to-Point

One producer, one consumer. The atom.

Wrapper around crossbeam::channel. Two newtypes:

  • QueueSender — wraps Sender
  • QueueReceiver — wraps Receiver

Constructors:

  • queue_bounded(capacity) — bounded channel. Sender blocks when full (backpressure).
  • queue_unbounded() — unbounded channel. No backpressure.

Methods on sender:

  • send(value: T) — blocks if bounded and full. Returns SendError if receiver dropped.

Methods on receiver:

  • recv() — blocking receive. Returns RecvError::Disconnected when sender dropped and queue empty.
  • try_recv() — non-blocking. Returns crossbeam::TryRecvError directly (not wrapped — composing programs like cache already have crossbeam in scope).
  • inner() — pub(crate), exposes the underlying &Receiver. Used by Mailbox and cache for crossbeam::Select across multiple receivers.

Error types: SendError and RecvError — both custom, both implement Display. Purpose is to decouple the public API from crossbeam's error types.

Tests (4): send/receive one, multiple in order, bounded backpressure (sender blocks in spawned thread until consumer frees a slot), shutdown on sender drop.

Topic — Fan-Out

One producer, N consumers. A proxy that writes to N queues.

How it works:

  1. Caller provides a Vec> — the write ends of N already-created queues. The kernel created these queues and gave the read ends to subscriber programs. The programs see plain queues; they don't know a topic exists.
  2. topic(capacity, outputs) creates:
  • An internal bounded input queue (in_tx, in_rx).
  • A spawned thread that loops: recv() from input, send(msg.clone()) to each output.
  1. Returns (TopicSender, TopicHandle).

TopicSender — wraps the input QueueSender. Same .send() interface as a queue. Writing here clones to all subscribers.

TopicHandle — holds the JoinHandle<()> for the fan-out thread. The thread exits when the producer drops TopicSender (input queue disconnects). When the thread exits, all output QueueSenders are dropped, cascading disconnection to every subscriber's receiver.

Backpressure: Bounded send to each subscriber. One slow subscriber stalls all others. Intentional — backpressure propagation.

Bounds: T: Send + Clone + 'static. Clone is required because the message is cloned to each subscriber.

Tests (3): one message reaches all subscribers, N messages in order per subscriber, shutdown cascades on sender drop.

Mailbox — Fan-In

N producers, one consumer. A proxy that reads from N queues.

How it works:

  1. Caller provides a Vec> — the read ends of N already-created queues. The kernel created these queues and gave the write ends to producer programs. The programs see plain queues; they don't know a mailbox exists.
  2. mailbox(inputs) creates:
  • An internal unbounded output queue (out_tx, out_rx).
  • A spawned thread that uses crossbeam::channel::Select to wait on all input receivers simultaneously.
  1. Returns MailboxReceiver — wraps the output QueueReceiver.

Fan-in thread logic:

  • Maintains a Vec> called alive.
  • Each iteration: builds a fresh Select over all alive receivers, calls select(), identifies which receiver fired via oper.index(), receives the message, forwards to output queue.
  • If a receiver returns Err (disconnected), that receiver is removed from alive.
  • Loop breaks when alive is empty (all inputs disconnected).
  • When the thread exits, out_tx drops, and the consumer sees Disconnected.

Partial disconnect resilience: If some producers drop but others remain, the mailbox continues receiving from the living ones. Only when all are gone does it signal disconnection.

Bounds: T: Send + 'static. No Clone needed — messages are moved, not copied.

MailboxReceiver — wraps QueueReceiver. Same interface as queue receiver:

  • recv() — blocking.
  • try_recv() — non-blocking, returns crossbeam::TryRecvError directly.
  • inner() — pub(crate), exposes underlying &Receiver for composing programs.

Tests (4): multiple senders one receiver, interleaved messages from different threads, shutdown on all senders dropped, partial sender drop still works.

How Concurrency Is Achieved Without Mutex

All three primitives use crossbeam::channel as their sole concurrency mechanism.

crossbeam::channel is a lock-free multi-producer multi-consumer channel built on atomic operations. The internal implementation uses compare-and-swap and spin-locks — no std::sync::Mutex anywhere.

The concurrency pattern:

  • Ownership transfer via types. QueueSender and QueueReceiver are separate types that can be moved between threads. Rust's type system enforces that the producer and consumer sides are distinct.
  • Threads communicate by sending messages, not by sharing memory. No shared mutable state exists within any of these primitives. Each spawned thread (topic fan-out, mailbox fan-in) owns its inputs exclusively.
  • Shutdown by disconnection. Dropping a sender/receiver propagates through the channel as a Disconnected error. No explicit coordination, no Arc> shutdown flags.
  • Backpressure via bounded channels. The topic uses bounded inputs; when full, the sender blocks naturally. The mailbox uses an unbounded output — it never stalls producers.

One subtle design choice: QueueReceiver::inner() and MailboxReceiver::inner() expose the raw crossbeam Receiver as pub(crate). This allows composing code (the cache, which uses crossbeam::Select) to treat queue receivers and mailbox receivers uniformly in a multi-channel select — without pulling the internal implementation up through the public API.
