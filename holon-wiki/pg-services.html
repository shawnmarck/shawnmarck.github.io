Services — Queue, Topic, and Mailbox EXPLAIN

TL;DR

Holon's messaging layer provides three primitives built on crossbeam::channel: Queue (1→1 point-to-point), Topic (1→N fan-out), and Mailbox (N→1 fan-in). All concurrency is achieved through message passing — zero Mutex. In the wat language, these appear as values created by make-pipe, make-topic, and select, with no special declarations needed. Shutdown propagates automatically by dropping senders/receivers.

All three use crossbeam::channel as their sole concurrency primitive — a lock-free MPSC channel built on atomic compare-and-swap operations (T0-016, §"How Concurrency Is Achieved Without Mutex").

Queue — Point-to-Point

The atom of the system. One producer, one consumer.

Rust types:

  • QueueSender — wraps crossbeam::Sender
  • QueueReceiver — wraps crossbeam::Receiver

Constructors:

  • queue_bounded(capacity) — bounded channel; sender blocks when full (backpressure)
  • queue_unbounded() — unbounded channel; no backpressure

Sender API: send(value: T) — blocks if bounded and full; returns SendError if receiver dropped.

Receiver API:

  • recv() — blocking; returns RecvError::Disconnected when sender dropped and queue empty
  • try_recv() — non-blocking; returns raw crossbeam::TryRecvError
  • inner() — pub(crate) raw &Receiver, used by Mailbox and the cache for crossbeam::Select

Error types (SendError, RecvError) are custom wrappers that decouple the public API from crossbeam's types (T0-016, §Queue).

wat layer: (make-pipe :capacity N :carries T) returns (tx rx). Send via (send tx value), receive via (recv rx) or (try-recv rx). "Destructure at creation" (T1-001, §Pipes). Pipes are values — "no defpipe, no defprocess, no defservice" (T1-001, §Design principle).

Topic — Fan-Out (1→N)

One producer, N consumers. A proxy that writes to N queues by cloning each message.

How it works (T0-016, §Topic):

  1. Caller provides Vec> — write ends of queues whose read ends belong to subscriber programs. Subscribers see plain queues; they don't know a topic exists.
  2. topic(capacity, outputs) creates an internal bounded input queue and spawns a fan-out thread that loops: recv() from input, send(msg.clone()) to each output.
  3. Returns (TopicSender, TopicHandle).

TopicSender — wraps the input QueueSender. Same .send() interface. Writing here clones to all subscribers.

TopicHandle — holds the JoinHandle<()> for the fan-out thread. When the producer drops TopicSender, the input disconnects, the thread exits, and all output QueueSenders are dropped — cascading disconnection to every subscriber.

Bounds: T: Send + Clone + 'static. Clone is required because the message is cloned to each subscriber.

Backpressure: Bounded send to each subscriber. One slow subscriber stalls all others — intentional backpressure propagation (T0-016, §Topic).

wat layer: (make-topic) wires 1→N fan-out over existing pipes. The wat spec notes this is synchronous, no thread, no queue at the specification level — "The caller's send IS the fan-out" (T1-001, §Topics). The Rust implementation adds the thread and internal queue for execution.

Mailbox — Fan-In (N→1)

N producers, one consumer. A proxy that reads from N queues via crossbeam::Select.

How it works (T0-016, §Mailbox):

  1. Caller provides Vec> — read ends of queues whose write ends belong to producer programs. Producers see plain queues; they don't know a mailbox exists.
  2. mailbox(inputs) creates an internal unbounded output queue and spawns a fan-in thread using crossbeam::channel::Select.
  3. Returns MailboxReceiver.

Fan-in thread logic:

  • Maintains a Vec> called alive.
  • Each iteration: builds a fresh Select over all alive receivers, calls select(), identifies which fired via oper.index(), receives the message, forwards to output.
  • Disconnected receivers are removed from alive.
  • Loop breaks when alive is empty — all inputs disconnected. Output sender drops; consumer sees Disconnected.

Partial disconnect resilience: If some producers drop but others remain, the mailbox continues receiving from the living ones. Only when all are gone does it signal disconnection (T0-016, §Mailbox).

Bounds: T: Send + 'static. No Clone needed — messages are moved, not copied.

MailboxReceiver — wraps QueueReceiver with the same interface: recv(), try_recv(), inner().

wat layer: (select rx1 rx2 ...) multiplexes N receive ends → (Some (idx value)) or :closed. "Parks when idle — zero CPU" (T1-001, §Pipes). This is the wat-level expression of what the Rust Mailbox implements as a dedicated composing thread.

Design Principles

Concurrency without Mutex

"Threads communicate by sending messages, not by sharing memory" (T0-016). The pattern:

  • Ownership transfer via types. QueueSender and QueueReceiver are separate, moveable types. Rust's type system enforces producer/consumer separation.
  • Shutdown by disconnection. Dropping a sender/receiver propagates as Disconnected. No explicit coordination, no Arc> shutdown flags.
  • Backpressure via bounded channels. Topics use bounded inputs; mailboxes use unbounded outputs (never stall producers).

Everything is a Value

In wat, pipes, topics, and processes are all plain values created by regular functions — not special declarations (T1-001, §Design principle):

"No defpipe. No defprocess. No defservice. Pipes are values. Topics are values. Processes are functions."

The seven concurrency forms are: make-pipe, send, recv, try-recv, select, make-topic, spawn, join.

Composition via inner()

QueueReceiver::inner() and MailboxReceiver::inner() expose the raw crossbeam Receiver as pub(crate), allowing composing code (e.g., the cache) to treat queue receivers and mailbox receivers uniformly in a multi-channel crossbeam::Select — without leaking the internal implementation through the public API (T0-016, §"One subtle design choice").
