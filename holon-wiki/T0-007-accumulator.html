T0-007: Accumulator HOW

Weighted vector accumulation used by the Reckoner.

Source: holon-rs/src/kernel/accumulator.rs (243 lines of production code)

What It IS#

A frequency-preserving streaming accumulator over bipolar vectors. It maintains a running sum of i8-valued Vector observations in a continuous f64 buffer (ndarray::Array1<f64>), along with a count of how many examples have been added.

The module-level doc states the design intent plainly:

"Accumulators are essential for building baselines from streaming data where frequency information matters. Unlike simple bundling, accumulators preserve the fact that a pattern seen 100 times should contribute 100x more than a pattern seen once."

The critical contrast with bundling is drawn explicitly:

"The critical difference between accumulator and bundle:

  • bundle([a, a, a, a, a]) = a (idempotent)
  • accumulator.add(a) * 5 preserves that a was seen 5 times"

This frequency preservation matters for anomaly detection: "common patterns should dominate the baseline, so rare anomalies have low similarity."

Internal Structure#

pub struct Accumulator {
    sums: Array1<f64>,   // Running sum of all vectors (not thresholded)
    count: usize,         // Number of examples accumulated
}

Two fields. The sums array lives in ndarray::Array1<f64> so that "decay, merge, and norm operations compile down to vectorized (SIMD) loops." The count is a plain usize.

How It Works#

Construction and Addition#

  • new(dimensions: usize) — creates an accumulator with zero-initialized sums and count zero.
  • add(&mut self, example: &Vector) — asserts dimension match, then adds each component of the example (cast i8 → f64) into sums. Increments count. The doc notes: "Unlike bundling, this does NOT threshold after each addition, preserving frequency information."
  • add_weighted(&mut self, example: &Vector, weight: f64) — same as add but each component is multiplied by weight before accumulating. Still increments count by 1.

Prototype Extraction#

Two ways to extract a prototype vector from the accumulated sums:

  1. normalize(&self) -> Vector — computes the L2 norm (sqrt(sums · sums)), returns zero vector if norm < 1e-10, otherwise divides each component by norm and thresholds back to bipolar {-1, 0, 1}. The doc: "Returns a continuous f64 vector suitable for cosine similarity." (Note: despite the doc comment, the actual return type is Vector with i8 components — the continuous values are quantized to bipolar via sign thresholding.)

  2. normalize_f64(&self) -> Vec<f64> — same normalization but returns the continuous f64 values without quantization.

  3. threshold(&self) -> Vector — sign-thresholds the raw sums to {-1, 0, 1} without normalizing. "This loses frequency information but produces a clean bipolar vector."

  4. raw_sums(&self) -> &[f64] — direct access to the internal f64 array.

Decay#

decay(&mut self, factor: f64) — multiplies every element of sums by factor. This is exponential decay. The doc: "Useful for time-weighted baselines where recent patterns should have more influence." The test confirms the mechanics: after adding [1,-1,1,-1] twice (sums = [2,-2,2,-2]), a decay of 0.5 yields [1,-1,1,-1].

Merge#

merge(&mut self, other: &Accumulator) — adds the other's sums elementwise into self.sums and adds the other's count. The doc: "Useful for parallel processing: accumulate in separate threads, then merge results."

Clear#

clear(&mut self) — zeros the sums array and resets count to zero.

Diagnostic Methods#

Three methods provide introspection into the state of the accumulator:

Capacity#

capacity(&self, codebook_size: usize) -> f64 — estimates remaining capacity as a fraction in [0.0, 1.0] (1.0 = empty, 0.0 = saturated). Uses the capacity bound N ≤ d / (2·ln(k)) where d is dimensionality and k is codebook size. The method computes max_items = d / (2·ln(k)), estimates current item count from L2²/d, and returns (1 - estimated/max) clamped to [0, 1].

Purity#

purity(&self) -> f64 — a "quantum-inspired purity measure" analogous to Tr(ρ²). Computed as min(1.0, d / L2²) where L2² is the squared L2 norm of sums. A single bipolar vector yields purity ≈ 1.0; N dissimilar accumulated vectors yield purity ≈ 1/N. The test asserts a single vector in 1024 dims has purity > 0.95, and 50 random vectors have purity < 0.5.

Participation Ratio#

participation_ratio(&self) -> f64 — computes PR = (Σ vᵢ²)² / Σ vᵢ⁴, a measure of how many dimensions carry significant energy. A single bipolar vector of dimension d yields PR ≈ d; energy concentrated in fewer dimensions reduces PR. Test: single bipolar in 1024 dims gives PR > 900; energy in only 10 dims gives PR < 20.

Usage in the Reckoner#

The Accumulator is the core storage primitive inside the Reckoner (both modes):

  • Discrete mode: one Accumulator per label (Vec<Accumulator> in ReckMode::Discrete).
  • Continuous mode: each ScalarBucket holds an Accumulator plus a center value for that bucket's scalar range.

The Reckoner's reckoner.rs imports it directly: use crate::kernel::accumulator::Accumulator;.

What It Is NOT#

  • It does not implement bundling or threshold after each addition — that is the explicit design distinction.
  • It does not store individual observations — only the running sum and count.
  • It does not have any notion of time indexing or ordering beyond the user-controlled decay() call.
  • The count field is incremented once per add() / add_weighted() call regardless of weight, so it tracks call count, not weighted sample mass.
