T0-014 — ExitObserver (Rust) — Verification

Source: holon-lab-trading/src/domain/position_observer.rs (291 lines)

Context: None. Read only what the code says.

What It Is

A PositionObserver. The module docstring calls it "position observer" (not "exit observer"). The file-level comment says: "Estimates position distances. Learned. Two continuous reckoners (trail, stop). Compiled from wat/position-observer.wat."

It has six fields. It is a struct, not a trait.

How It Predicts Distances

One public prediction method: reckoner_distances(position_thought: &Vector) -> Option.

  1. Check experience of both reckoners (trail and stop). If either has zero experience, return None.
  2. Query trail_reckoner.query(position_thought) → trail distance.
  3. Query stop_reckoner.query(position_thought) → stop distance.
  4. Return Some(Distances { trail, stop }).

The broker owns the full cascade (reckoner → accumulator → default). This method is Tier 1 only — raw reckoner output, no fallback.

A second method recommended_distances exists but is #[cfg(test)] only. It implements the full cascade: reckoner → broker accumulator → default_distances. Not used in production.

Both methods query on position_thought only (Proposal 026). No composition with market thought at query time.

How It Learns

One learning method: observe_distances(position_thought: &Vector, optimal: &Distances) -> (f64, f64, f64, f64, f64, f64).

  1. Pre-learning prediction: Query each reckoner for what it would predict given current state. If no experience, use default.
  2. Learn: Call trail_reckoner.observe_scalar(position_thought, optimal.trail, 1.0) and stop_reckoner.observe_scalar(position_thought, optimal.stop, 1.0). Weight is always 1.0 — each observation counts once. The value (optimal distance) goes to the bucket center; the thought goes to the prototype. Weight does NOT scale by magnitude.
  3. Return: (trail_error, stop_error, predicted_trail, predicted_stop, optimal_trail, optimal_stop).

The "optimal" distances come from hindsight — simulation sweeps candidate values against actual price histories to determine what would have been ideal (defined in simulation module, not here).

Noise Subspace

strip_noise(thought: &Vector) -> Vector — converts to f64, runs noise_subspace.anomalous_component(), converts back. Returns what the subspace cannot explain. Same pattern as market_observer.

Discrepancy With Module Docstring

Lines 1–5 state: "Intentionally simpler than MarketObserver. No noise-subspace, no curve, no engram gating."

The struct actually has a noise_subspace field (line 27–28) and a strip_noise method (line 165). The docstring contradicts the code.

Tests

7 tests in #[cfg(test)] mod tests:

  • Construction defaults correct
  • Returns defaults when empty
  • experienced() is false initially, true after one observation
  • Reckoner cascade produces non-zero values after 10 training observations
  • Two reckoners are independent, both continuous, different names
  • strip_noise returns correct-dimension vector
  • Noise subspace initialized and functional

All use DIMS=4096, RECALIB=100, DEFAULT_TRAIL=0.02, DEFAULT_STOP=0.05.

Summary

PositionObserver predicts two continuous values (trailing stop distance, safety stop distance) from a position thought vector using two independent continuous Reckoners. It learns from hindsight-optimal distances with unit weight. It has a noise subspace despite its own docstring claiming it doesn't. The broker owns the production cascade; the observer only exposes raw reckoner queries.
