T0-006 · Walkable — the AST traversal trait and thought expression tree

What it IS

Walkable is the trait that enables zero-serialization encoding: any Rust type implementing Walkable can be directly converted into high-dimensional vectors without passing through JSON or EDN. It defines a structural type system (WalkType) that the encoder recursively walks to produce role-filler bindings, which are then bundled into a single vector.

Source: holon-rs/src/kernel/walkable.rs (1002 lines)

Also relevant: holon-rs/src/kernel/encoder.rs — contains the encode_walkable recursive traversal.

"This module provides the Walkable trait that allows encoding in-memory data structures directly without JSON/EDN serialization."

The structural type system — WalkType

There are five structural types:

WalkTypeShapeEncoder semantics
ScalarSingle atomic valueLooked up via VectorManager or special numeric encoding
MapKey-value pairs (ordered list of (String, WalkableValue))Role-filler binding: each field's key (role) is bound to its value's vector (filler), then all bindings are bundled
ListOrdered sequencePosition-encoded: each element bound to its positional path [0], [1], … then bundled
SetUnordered unique itemsItems bundled directly, then bound with a "set_indicator" role vector
SpreadOrdered sequence with fan-outEach element fans out into its own independent leaf binding with its own indexed path. Used when per-element attribution matters.
"List and Set are composed: items are encoded and bundled into a single vector. Spread is fanned out: each element becomes an independent leaf binding with its own indexed path (e.g., field.[0], field.[1]). The Walkable implementation chooses which semantics to use per field."

WalkableValue — the AST node enum

WalkableValue is the owned enum that represents the tree. It is the recursive data structure the encoder traverses:


pub enum WalkableValue {
    Scalar(ScalarValue),
    Map(Vec<(String, WalkableValue)>),
    List(Vec),
    Set(Vec),
    Spread(Vec),
}

Each variant is recursive (except Scalar which is a leaf). This is the thought expression tree: structured data decomposed into a form the encoder can recursively walk.

Scalar types — ScalarValue

The leaf nodes of the tree. Eight variants:

VariantEncoding path
String(String)Atom lookup via to_atom() → VectorManager.get_vector()
Int(i64)Atom lookup: i64.to_string()
Float(f64)Atom lookup: f64.to_string()
Bool(bool)Atom lookup: "true" / "false"
NullAtom lookup: "null"
LogFloat { value, scale }Log10(value) → positional encoding. Equal ratios = equal similarity drops.
LinearFloat { value, scale }Direct positional encoding. Equal differences = equal similarity drops.
TimeFloat { value, resolution }Circular (hour-of-day, day-of-week, month) + positional. Temporally close events produce similar vectors.
"Numeric scalars don't use atom encoding — handled specially by encoder."

The three numeric scalars (LogFloat, LinearFloat, TimeFloat) are flagged by is_numeric_scalar() and routed to special encoder methods instead of atom lookup. Their to_atom() methods produce prefixed strings ("$log:…", "$linear:…", "$time:…") but these are informational only — the encoder dispatches on the variant, not on the atom string.

Default scales and resolutions

  • ScalarValue::log(value) — scale defaults to 1000.0
  • ScalarValue::linear(value) — scale defaults to 1000.0
  • ScalarValue::time(value) — resolution defaults to TimeResolution::Hour

The Walkable trait


pub trait Walkable {
    fn walk_type(&self) -> WalkType;
    fn walk_scalar(&self) -> ScalarValue { panic!(…) }
    fn walk_map_items(&self) -> Vec<(&str, WalkableValue)> { panic!(…) }
    fn walk_list_items(&self) -> Vec { panic!(…) }
    fn walk_set_items(&self) -> Vec { panic!(…) }
    fn walk_spread_items(&self) -> Vec { panic!(…) }
    fn to_walkable_value(&self) -> WalkableValue { … }
    // Zero-allocation fast path:
    fn walk_scalar_ref(&self) -> ScalarRef<'_> { … }
    fn walk_map_visitor(&self, &mut dyn FnMut(&str, WalkableRef<'_>)) { … }
    fn walk_list_visitor(&self, &mut dyn FnMut(WalkableRef<'_>)) { … }
    fn walk_set_visitor(&self, &mut dyn FnMut(WalkableRef<'_>)) { … }
    fn has_fast_visitor(&self) -> bool { false }
}

Dispatch rule: implement walk_type() to declare your structural type, then implement the corresponding accessor. Calling the wrong accessor panics with a diagnostic message.

"Based on walk_type(), implement the corresponding method: WalkType::Scalar → walk_scalar(), WalkType::Map → walk_map_items(), WalkType::List → walk_list_items(), WalkType::Set → walk_set_items()."

Two encoding paths

  1. Owned path: walk_map_items() etc. return Vec — allocates, but simpler to implement.
  2. Zero-allocation visitor path: walk_map_visitor() etc. call a closure with WalkableRef<'_> — borrows data, avoids cloning strings. The encoder checks has_fast_visitor() and uses this path when available.

WalkableRef — the borrowed counterpart


pub enum WalkableRef<'a> {
    Scalar(ScalarRef<'a>),   // zero-allocation for flat scalars
    Nested(WalkableValue),    // owned allocation for nested structures
}

ScalarRef<'a> mirrors ScalarValue but borrows strings (&'a str instead of String). Constructor helpers: WalkableRef::string(s), WalkableRef::int(i), WalkableRef::float(f), WalkableRef::bool(b), WalkableRef::null(), WalkableRef::time(value), WalkableRef::nested(value).

How encode() recursively walks the tree

The encoder's encode_walkable(walkable) calls encode_walkable_recursive(walkable, None), which dispatches on walkable.walk_type():

Map encoding (WalkType::Map)


encode_walkable_map(walkable, prefix):
  if has_fast_visitor() → encode_walkable_map_fast (visitor-based)
  else → walk_map_items() → for each (key, value):
    key_path = make_path(prefix, key)       // e.g. "name" or "parent.name"
    role_vec  = encode_atom(key_path)       // VectorManager lookup
    filler_vec = encode_walkable_value_recursive(value, Some(key_path))
    bound = bind(role_vec, filler_vec)      // element-wise multiply
    push bound
  bundle(all bounds)                        // element-wise sign (majority)
"The key insight is role-filler binding: we bind each value (filler) with its key (role) before bundling."

List/Spread encoding (WalkType::List or WalkType::Spread)

Both use identical encoding logic:


encode_walkable_list(walkable, prefix):
  walk_list_items() → for each (i, item):
    pos_prefix = make_path(prefix, "[i]")   // e.g. "items.[0]"
    item_vec  = encode_walkable_value_recursive(item, Some(pos_prefix))
    pos_vec   = encode_atom(pos_prefix)
    bound = bind(pos_vec, item_vec)
    push bound
  bundle(all bounds)

The difference between List and Spread is not in the encoding but in the striped/fan-out path (see below).

Set encoding (WalkType::Set)


encode_walkable_set(walkable, prefix):
  set_indicator = encode_atom("set_indicator")
  walk_set_items() → for each item:
    item_vec = encode_walkable_value_recursive(item, prefix)
    push item_vec
  bundled = bundle(all item_vecs)
  bind(set_indicator, bundled)

Sets are order-independent: no positional binding. The set_indicator role vector distinguishes a set from a random bundle.

Recursive WalkableValue encoding

The encode_walkable_value_recursive method handles the full tree for nested values:

WalkableValue variantBehavior
Scalar(s)encode_scalar_value(s, prefix) — atom lookup or numeric encoding
Map(items)Role-filler bind each entry, then bundle (same as above)
List(items)Position-encode each element, then bundle
Set(items)Bundle items, then bind with set_indicator
Spread(items)Position-encode each element, then bundle (identical to List)

Empty container encoding

  • Empty Map → encode_atom(make_path(prefix, "{}"))
  • Empty List → encode_atom(make_path(prefix, "[]"))
  • Empty Set → encode_atom(make_path(prefix, "#{}"))

Striped encoding and the difference between List and Spread

In the hierarchical encode path (encode_walkable), List and Spread are encoded identically. The difference appears in the striped path (encode_walkable_striped):

collect_leaf_bindings treats them differently:

  • List: Produces a single leaf binding — the positionally-encoded bundle is one role-filler pair assigned to one stripe.
  • Spread: Each element becomes an independent leaf binding at its own indexed path (field.[0], field.[1], …), each independently hashed to a stripe.
"Spread fans out — each element becomes an independent leaf binding with its own indexed path."
"Use when per-element attribution or rule crafting is needed."

Built-in Walkable implementations

The file provides Walkable impls for:

  • Scalars: String, &str, i64, i32, u32, u64, usize, f64, f32, bool
  • Containers: Vec, [T], HashMap, HashMap<&str, V>, HashSet, BTreeSet, Option
  • Self-referential: WalkableValue itself
  • JSON bridge: serde_json::Value — Null/Bool/Number/String → Scalar, Array → List, Object → Map

Structural forms in wat (orthogonal context)

The wat/core/structural.wat file defines three structural forms orthogonal to the vector algebra:

"A struct cannot be bound, bundled, or cosined. It holds things that can. An enum variant is an identity, not a geometric object. A protocol is a contract, not a computation."
Formwat syntaxRust target
Product (struct)(struct name field1 field2 …)Rust struct with named fields. Projection via (:field record). Functional update via (update record :f1 v1 …).
Coproduct (enum)(enum name variant1 variant2 …)Rust enum. Exhaustive match required; wildcard _ forbidden on closed enums.
Type class (defprotocol)(defprotocol name (fn-name [params] …))Rust trait. Satisfiability via (satisfies struct-name protocol-name :fn1 impl1 …).

These forms organize programs but do not enter the vector algebra. A Walkable impl bridges between a struct and the encoder, but the struct itself is not a vector object.
