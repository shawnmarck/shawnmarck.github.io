T1-003: wat Proposals System DESCRIBE

Verified: 2026-04-14 Source: /root/projects/holon-workspace/wat/docs/proposals/ Correction: The system contains 18 proposals (15 in 2026/03, 3 in 2026/04), not 31 as stated in the task. Each proposal has exactly 4 files: PROPOSAL.md, RESOLUTION.md, review-beckman.md, review-hickey.md — 72 files total.

1. The Proposal Process#

How a proposal is created#

A proposal follows a fixed structure in its PROPOSAL.md:

  1. Scope — classified as core, structural, or follows from a prior proposal
  2. The Current State — what exists today, with code examples from the enterprise codebase or the wat runtime
  3. The Problem — concrete gap, often discovered by automated "wards" (gaze, forge, sever, scry, reap) or by tracing type violations between wat specs and Rust implementation
  4. The Proposed Change — options with code examples, usually 2-3 alternatives
  5. The Algebraic Question — does this compose with the existing monoid (bind/bundle)? Does it introduce new algebraic structure? Every proposal must address this.
  6. The Simplicity Question — Hickey's test: "Is this simple or easy?" What is being complected? Could existing primitives solve it?
  7. Questions for Designers — specific yes/no and tradeoff questions for the review panel

How proposals are reviewed#

Each proposal is reviewed independently by two designers: Brian Beckman and Rich Hickey. Reviews are written as separate markdown files (review-beckman.md, review-hickey.md). Each review states a verdict:

  • ACCEPTED — as proposed
  • CONDITIONAL — accepted with specific conditions
  • REJECTED — with reasoning

The reviews are not rubber stamps. They have reversed proposals (001), overruled each other, and been overruled by a "datamancer" role that synthesizes final decisions (see 013, 003/003).

How proposals are resolved#

The RESOLUTION.md synthesizes the two reviews into a final decision:

  • ACCEPTED — with specific form syntax, placement, and constraints
  • REJECTED — with open questions forwarded to future proposals
  • ACCEPTED with tension — accepted but a specific disagreement is deferred
  • SUPERSEDED — the debate revealed a simpler answer (see 04/003)

Resolution often involves overriding one designer. The resolution document records who was overridden and why. Beckman tends toward categorical framing (product/coproduct/co-algebra). Hickey tends toward simplicity and "don't add what functions already provide."

2. The Panel System#

Brian Beckman#

  • Role: Algebraic authority. Frames proposals in category theory — monoids, coalgebras, functors, catamorphisms, universal constructions.
  • Pattern: Asks "does this compose?" and "what categorical construction is this?" Tends to see structure where Hickey sees convention.
  • Key moves: Proposed fold as the catamorphism (001 review, accepted as 002). Framed protocols as "type classes — the third construction alongside struct (product) and enum (coproduct)" (014). Argued journal operations are co-generators of a coalgebra and belong in core (004).
  • Overridden: On struct field types (007 — wanted mandatory types, Hickey's untyped won). On defservice (04/003 — the datamancer showed functions + select suffice).

Rich Hickey#

  • Role: Simplicity authority. Applies "simple made easy" — distinguishes simple (one concept, one job) from easy (familiar, convenient). Protects the six primitives from growth.
  • Pattern: Asks "does this complect?" and "could existing forms do this?" Tends to reject new forms when discipline or data restructuring suffices.
  • Key moves: Rejected defprocessor flatly — "the function shape IS the declaration" (001). Won on untyped struct fields — "types are documentation, not enforcement" (007). Won keyword-as-function access — "keywords read like English" (007). Argued fold is a caller concern, not a callee declaration (002 — partially reversed when fold was accepted as control form).
  • Overridden: On journal operations in stdlib (004 — reversed after opacity argument). On nil (013 — datamancer chose structural absence, no nil value). On channels (003 — channels deleted as contradicting fold-based design).

The Datamancer#

Not a formal reviewer but an arbitrating role visible in resolutions. Appears when the two designers disagree or when neither option is right. Overruled both designers on boolean/nil (013 — chose "structural absence" with ? field convention, neither Clojure truthiness nor Rust separation). Overruled both on defservice (04/003 — showed that select + functions + spawn subsume defprocess and defservice entirely).

3. Key Design Decisions Across Proposals#

Principles that emerge#

  1. Transparency to Rust. Wat compiles to Rust. Form names should match Rust (struct, not defstruct). Access syntax should map obviously (:field record → record.field). The wat spec should be readable alongside the implementation.

  2. Protect the algebra. The six Primitives (atom, bind, bundle, cosine) and journal coalgebra (journal, observe, predict, decay, curve) are the foundation. Proposal 001 was rejected to prevent a seventh primitive. Only fold was added — as a control form, not a primitive.

  3. Simple > easy. Every proposal asks "is this simple or easy?" Convenience is rejected if it complected concerns. Variadic update (012) accepted because it completes an existing form. defprocessor (001) rejected because it packaged four concerns into one form.

  4. Types are documentation, not enforcement. Structs are untyped (007). Protocols check arity, not return types (014). when without else generates Option<T> — the compiler infers, the specifier doesn't annotate (013).

  5. Opacity creates boundaries. If a type is opaque, its interface operations are core (004). This rule determined that observe/predict/decay are core, not stdlib.

  6. The four-layer model (003): Syntax (compiler builtins), Core (algebra), Stdlib (derived operations), Userland (domain code). Each layer has a boundary rule — stdlib cannot import from userland, core can only be extended by proposal.

Forms added to the language#

Form Proposal Layer Notes

send, recv, try-recv, select, spawn 04/002, 04/003 Syntax (concurrency) Pipe verbs. select absorbs select-ready.

Forms rejected or retired#

  • defprocessor — rejected (001). The function shape (state, event) → state IS the declaration.
  • std/channels.wat — deleted (003). Contradicted fold-based design.
  • (field ...) from proposal 011 — partially superseded by protocols (014 resolution says protocols subsume derived fields). However, 011 was accepted before 014, so field lives in std/fields.wat with uncertain long-term status.
  • defprocess, defservice — withdrawn (04/003). Functions + select + spawn suffice.
  • nil — rejected (013). Absence is structural (when not executing), not a value.

Temporal pattern#

The March 2026 proposals (001-015) built the language foundation: fold, layers, core boundaries, all of the Structural Forms, and the stdlib seeds. The April 2026 proposals (04/001-003) extended into concurrency — parallelism, pipes, and service patterns. The April proposals show the system maturing: the datamancer overruled both designers on 04/003, finding that the simplest answer (functions + select) was more powerful than either proposed option.
