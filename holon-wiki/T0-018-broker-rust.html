T0-018: Broker (Rust) — Context-Free Verification WHAT

Source: src/domain/broker.rs (146 lines, 70 production + 76 test) + src/programs/app/broker_program.rs (207 lines) Verdict: Accountable. Complete. Stripped to essentials. Quoted below.

1. Module Purpose — broker.rs (Lines 1–6)#

/// broker.rs — The accountability primitive. Binds one market observer +
/// one position observer. The broker owns the game — gate 4, anxiety
/// atoms, exit/hold decisions. The treasury owns papers and capital.

What it is: The broker binds one market observer to one position observer. It is the accountability unit. It owns outcome counts, a grace-rate gate, and an active direction. It does NOT own paper trades, accumulators, or dollar P&L — the treasury owns papers and capital. The broker records outcomes directly and encodes anxiety atoms.

2. Imports (Line 5)#

Import From Purpose
Direction, Outcome crate::types::enums Up/Down direction, Grace/Violence outcome

That's it. One import. The broker is self-contained.

4.2 market_idx() / position_idx() — Grid Derivation (Lines 47–54)#

pub fn market_idx(&self) -> usize {
    self.slot_idx / self.position_count
}
pub fn position_idx(&self) -> usize {
    self.slot_idx % self.position_count
}

Semantics: Integer division and modulo. With position_count = M:

  • slot_idx = market_idx * M + position_idx
  • Therefore: market_idx = slot_idx / M, position_idx = slot_idx % M

Test confirmation (line 99–101): slot_idx=5, position_count=3 → market_idx=1, position_idx=2.

4.3 gate_open() — The Gate (Lines 58–61)#

pub fn gate_open(&self) -> bool {
    let cold_start = self.grace_count < 50 || self.violence_count < 50;
    cold_start || self.expected_value > 0.5
}

Two-phase gate:

  1. Cold start: If either outcome count is below 50, the gate is always open. Not enough data to judge.
  2. Warm: After ≥50 of both outcomes, the gate opens only when expected_value > 0.5. A broker must have a strict majority of Grace outcomes to keep trading.

Key detail from tests: expected_value = 0.5 is NOT sufficient — the threshold is > 0.5, not >= 0.5. At exactly 50% grace, the gate is closed (verified in test_gate_open_high_grace, line 128).

Previous behavior: The old gate used > 0.0 (any positive expected value). Now it requires > 0.5 (majority grace rate).

4.4 record_outcome() — Outcome Recording (Lines 64–71)#

pub fn record_outcome(&mut self, outcome: Outcome) {
    match outcome {
        Outcome::Grace => self.grace_count += 1,
        Outcome::Violence => self.violence_count += 1,
    }
    self.trade_count += 1;
    self.expected_value = self.grace_count as f64 / self.trade_count as f64;
}

What it does: Increments the appropriate outcome counter, increments trade_count, and recomputes expected_value as a simple grace rate.

What it doesn't do: No PropagationFacts returned. No scalar accumulator updates. No dollar P&L computation. No EMA smoothing. The method mutates the broker directly — no return value.

anxiety_atoms(receipt, current_candle, current_price) (Lines 40–66)#

Pure function computing 4 anxiety atoms per active receipt:

Atom AST Form Value Description
candles-remaining Bind(Atom, Log) (deadline - current_candle).max(1.0) Candles left. Floored at 1.0
time-pressure Bind(Atom, Linear) age / total_life (1.0 if zero lifespan) Fraction of position consumed
unrealized-residue Bind(Atom, Linear) (current_value - amount) / amount Percentage unrealized P&L
paper-age Bind(Atom, Log) (current_candle - entry_candle).max(1.0) Age in candles. Floored at 1.0

Where:

  • age = current_candle - receipt.entry_candle
  • total_life = receipt.deadline - receipt.entry_candle
  • current_value = receipt.units_acquired * current_price
  • unrealized = (current_value - receipt.amount) / receipt.amount

6.4 broker_program() — Main Loop (Lines 70–207)#

pub fn broker_program(
    chain_rx: QueueReceiver<MarketPositionChain>,
    cache: CacheHandle<ThoughtAST, Vector>,
    vm: VectorManager,
    scalar: Arc<ScalarEncoder>,
    console: ConsoleHandle,
    db_tx: QueueSender<LogEntry>,
    mut broker: Broker,
    treasury: TreasuryHandle,
) -> Broker

Parameters: Queue receiver for chains, encoding infrastructure, console, DB sender, the broker domain struct, and a treasury handle.

Returns: The trained Broker when the chain source disconnects.

Per-candle algorithm:

1. Receive MarketPositionChain from queue
2. Extract Direction from market_prediction, set broker.active_direction
3. Determine asset pair from Direction:
   - Up: source_asset → target_asset
   - Down: target_asset → source_asset
4. Submit paper via treasury.submit_paper(). If approved → push receipt
5. Iterate active_receipts, check state via treasury.get_paper_state():
   - Active → keep
   - Grace → broker.record_outcome(Grace), remove
   - Violence → broker.record_outcome(Violence), remove
   - None → remove
6. Encode anxiety atoms for all active receipts + active-positions count
7. Log BrokerSnapshot every 100 candles
8. Log PhaseSnapshot every candle (slot 0 only)
9. Emit telemetry metrics (total_ns, learn_grace, learn_violence, active_receipts)
10. Console diagnostic every 1000 candles

Telemetry dimensions: {"slot": broker.slot_idx}. Telemetry ID: "broker:{slot_idx}:{candle_count}".

8. Key Design Decisions#

Decision Detail
Grace rate, not EMA expected_value = grace_count / trade_count. Simple fraction. No smoothing, no dollar P&L.
Gate threshold > 0.5 Strict majority required. 0.5 exactly → gate closed. Previous: > 0.0.
Cold start = 50 Either outcome count < 50 → always open. Same threshold as before.
Treasury owns positions Broker submits papers and reads state. Does not own PaperEntry.
Anxiety atoms for gate 4 4 atoms per receipt encoded but gate 4 reckoner not yet built.
active_direction on broker Set every candle from market prediction. None only at construction.
Slot 0 emits PhaseSnapshot Avoids N×M duplicate phase writes. Only one broker logs phase data.
Returns broker on disconnect "The accounting comes home" — the trained broker struct is returned to the caller.

9. Findings#

  1. Complete. All 8 fields documented. All 4 domain methods documented. All broker_program functions documented. Every code path traced.
  2. No hidden mutation. The broker owns its state and mutates only itself. The broker_program mutates the broker via record_outcome() and active_direction.
  3. No shared observer references. The broker does not talk to observers. It does not produce PropagationFacts. Learning is recorded as counts.
  4. Gate has two distinct modes with clear boundary at 50 of each outcome. Strict threshold at > 0.5.
  5. Treasury interaction is via handle. TreasuryHandle uses per-broker mailbox channels — no shared mutation.
  6. Anxiety atoms are computed but not yet consumed. Gate 4 (Exit/Hold reckoner) is marked "to be built."
  7. 6 tests cover construction, grid math, cold-start gate, low-grace gate, high-grace gate (including strict threshold), and outcome recording.
  8. No broker_program.rs tests. The program is integration-level — tested via smoke tests (wat-vm.sh smoke 500).
