Programs — the concurrent pipeline architecture EXPLAIN

#TL;DR

A program in Holon is a function running on its own thread, communicating exclusively through <a class="wikilink" href="pg-services.html">queues</a>, <a class="wikilink" href="pg-services.html">topics</a>, and <a class="wikilink" href="pg-services.html">mailboxes</a>. There is no shared mutable state and zero Mutex. The <a class="wikilink" href="T1-001-wat-language.html">wat language</a> expresses this as spawn — "a process is a function" (T1-001, §Threads). At runtime, the <a class="wikilink" href="pg-wat-vm.html">wat-vm</a> wires N market observers, M position observers, and N×M brokers into a concurrent pipeline grid, where each stage produces a typed chain value that proves which stage produced it. Resource handles (console, cache, database) are pre-allocated into pools; every program thread claims exactly one of each, and orphan detection panics if any go unclaimed.

Wiki composite · Sources: T0-016 (services-rust), T0-017 (wat-vm-rust), T1-001 (wat-language), T0-013 (market-observer-rust), T0-014 (exit-observer-rust)

#What a Program Is

In wat, a program is a function launched with (spawn fn) → Handle, joined with (join handles) → list of return values. The design principle is stark:

"No defpipe. No defprocess. No defservice. Pipes are values. Topics are values. Processes are functions." (T1-001, §Design principle)

In Rust, this maps to: create input/output <a class="wikilink" href="pg-services.html">queues</a>, spawn std::thread::spawn, pass senders/receivers by move. The function owns its inputs exclusively; threads communicate by sending messages, not by sharing memory (T0-016, §"How Concurrency Is Achieved Without Mutex").

#The Three Program Types

The <a class="wikilink" href="pg-wat-vm.html">wat-vm</a> runs exactly three kinds of program threads (T0-017):

#Market Observer (market_observer_program)

Role: Receive candles, encode market features, predict direction.

Inputs: One bounded candle queue (capacity 1) from the main thread. Outputs: N <a class="wikilink" href="pg-services.html">topics</a> — one per position observer — carrying <a class="wikilink" href="pg-chain-types.html">MarketChain</a> values.

Each market observer holds a <a class="wikilink" href="pg-market-observer.html">MarketObserver</a> struct: a <a class="wikilink" href="pg-reckoner.html">Reckoner</a> for Up/Down prediction, a noise subspace for anomaly detection, a <a class="wikilink" href="pg-curve.html">curve</a> for conviction calibration, and an engram for gating low-conviction predictions (T0-013). It encodes the candle into a <a class="wikilink" href="pg-thought-system.html">ThoughtAST</a>, sends it to the <a class="wikilink" href="pg-encoding-cache.html">Encoding Cache</a>, receives the thought vector, strips noise, and predicts. Results are fanned out via topics to all position observers.

Per-stream default: N = 6 (one per <a class="wikilink" href="pg-lens.html">market lens</a>).

#Position Observer (position_observer_program)

Role: Receive market observations from N market observers, predict exit distances.

Inputs: N bounded queues — one from each market observer's topic output — assembled into PositionSlots. Outputs: One unbounded queue carrying <a class="wikilink" href="pg-chain-types.html">MarketPositionChain</a> values, consumed by a downstream broker.

Each position observer holds a <a class="wikilink" href="pg-exit-observer.html">PositionObserver</a> struct (also called "exit observer"): two continuous <a class="wikilink" href="pg-reckoner.html">Reckoner</a>s for trailing stop and safety stop distances (T0-014). It receives a <a class="wikilink" href="pg-chain-types.html">MarketChain</a> from each market observer, encodes its own position-specific features, and produces a <a class="wikilink" href="pg-chain-types.html">MarketPositionChain</a> that flattens all market fields plus adds position fields.

Per-stream default: M = 2 (one per <a class="wikilink" href="pg-lens.html">position lens</a>).

#Broker (broker_program)

Role: Receive position observations, manage simulated trades, propagate learn signals back.

Inputs: One <a class="wikilink" href="pg-chain-types.html">MarketPositionChain</a> queue from a position observer. Outputs (via unbounded queues → mailboxes):

  • ObsLearn → market observer's mailbox (direction feedback)
  • PositionLearn → position observer's mailbox (exit distance feedback)
  • TradeUpdate → position observer's trade mailbox (trade state)

The broker owns the production cascade: it decides when to enter/exit based on predictions and distances, then sends hindsight-optimal observations back through the learn queues so observers improve (T0-014, T0-017).

Per-stream count: N × M = 6 × 2 = 12 brokers.

#Data Flow Through the Grid

flowchart TD
    main["Main thread\nsend ObsInput"] --> mkt_obs["Market Observer i\n(thread)\nencode → predict"]
    mkt_obs -->|"Topic fan-out\nsend MarketChain"| pos0["Position Observer 0"]
    mkt_obs --> pos1["Position Observer 1"]
    mkt_obs --> posM["Position Observer M"]
    pos0 -->|"send MarketPositionChain"| broker["Broker [i, ei]\n(thread)"]
    pos1 --> broker
    posM --> broker
    broker -.->|"ObsLearn → market observer i's mailbox"| mkt_obs
    broker -.->|"PositionLearn → position observer ei's mailbox"| posM
    broker -.->|"TradeUpdate → position observer ei's trade mailbox"| posM


(T0-017, §"Data flow through the grid")


Each arrow is a <a class="wikilink" href="pg-services.html">Queue</a>. The fan-out from market to position observers uses <a class="wikilink" href="pg-services.html">Topic</a> (1→N). The learn-signal aggregation from multiple brokers back to one observer uses <a class="wikilink" href="pg-services.html">Mailbox</a> (N→1).

#Resource Pools and Orphan Detection


Console, cache, and database handles are pre-allocated into HandlePools. Each program thread claims exactly one of each. Pools assert finish() — if any handle remains unclaimed, the program panics (T0-017, §"Resources allocated from pools").


This is orphan detection: if a thread was supposed to be spawned but wasn't (or was wired incorrectly), the unclaimed handle triggers a panic at startup, preventing deadlocks from invisible missing consumers.




#Thread Count


For one stream with N market observers, M position observers (T0-017, §Thread count):


N + M + (N×M) + 3 threads minimum (database + cache + console drivers).


With defaults (N=6, M=2): 6 + 2 + 12 + 3 = 23 threads.




#Shutdown by Disconnection


Programs do not use explicit shutdown flags. Shutdown cascades through <a class="wikilink" href="pg-services.html">Queue</a> disconnection:


    

  1. Main thread drops candle_txs → market observers see Disconnected on input queues.
    2. 

  2. Market observer threads are joined (collected back with trained state).
    3. 

  3. TopicHandles are dropped → topics drain, then output senders drop → position observers see Disconnected.
    4. 

  4. Position observer threads are joined.
    5. 

  5. Broker input queues disconnect automatically → broker threads are joined.
    6. 

  6. Finally, cache and database drivers are joined (T0-017, §Shutdown sequence).
    7. 





"Dropping a sender/receiver propagates through the channel as a Disconnected error. No explicit coordination, no Arc<Mutex<bool>> shutdown flags." (T0-016)






#Concurrency Model


All three primitives — <a class="wikilink" href="pg-services.html">Queue</a>, <a class="wikilink" href="pg-services.html">Topic</a>, <a class="wikilink" href="pg-services.html">Mailbox</a> — use crossbeam::channel as their sole concurrency mechanism. The internal implementation uses atomic compare-and-swap operations; no std::sync::Mutex exists anywhere in the messaging layer (T0-016, §"How Concurrency Is Achieved Without Mutex").


The pattern:


    

  • Ownership transfer via types — QueueSender/QueueReceiver are separate, moveable types.
    • 

  • Threads communicate by sending messages, not by sharing memory — no shared mutable state within any primitive.
    • 

  • Shutdown by disconnection — drop propagates as Disconnected.
    • 

  • Backpressure via bounded channels — topics use bounded inputs; mailboxes use unbounded outputs.
    •
