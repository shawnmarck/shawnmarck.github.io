Programs — the concurrent pipeline architecture EXPLAIN

TL;DR#

A program in Holon is a function running on its own thread, communicating exclusively through queues, topics, and mailboxes. There is no shared mutable state and zero Mutex. The wat language expresses this as spawn — "a process is a function" (T1-001, §Threads). At runtime, the wat-vm wires N market observers, M regime observers, and N×M brokers into a concurrent pipeline grid, where each stage produces a typed chain value that proves which stage produced it. Resource handles (console, cache, database) are pre-allocated into pools; every program thread claims exactly one of each, and orphan detection panics if any go unclaimed.

Wiki composite · Sources: T0-016 (services-rust), T0-017 (wat-vm-rust), T1-001 (wat-language), T0-013 (market-observer-rust), T0-014 (exit-observer-rust)

What a Program Is#

In wat, a program is a function launched with (spawn fn) → Handle, joined with (join handles) → list of return values. The design principle is stark:

"No defpipe. No defprocess. No defservice. Pipes are values. Topics are values. Processes are functions." (T1-001, §Design principle)

In Rust, this maps to: create input/output queues, spawn std::thread::spawn, pass senders/receivers by move.

The Four Program Types#

The wat-vm runs exactly four kinds of program threads (T0-017):

Market Observer (market_observer_program)#

Role: Receive candles, encode market features, predict direction.

Inputs: One bounded candle queue (capacity 1) from the main thread. Outputs: N topics — one per regime observer — carrying MarketChain values.

Each market observer holds a MarketObserver struct: a Reckoner for Up/Down prediction, a noise subspace for anomaly detection, a curve for conviction calibration, and an engram for gating low-conviction predictions (T0-013). It builds indicator rhythm ASTs from the candle window via market_rhythm_specs(lens), encodes the market thought, strips noise, and predicts. Results are fanned out via topics to all regime observers.

Per-stream default: N = 6 (one per market lens).

Regime Observer (regime_observer_program)#

Role: Receive market observations from N market observers, compose regime-specific vocabulary facts and rhythm ASTs.

Inputs: N bounded queues — one from each market observer's topic output — assembled into RegimeSlots. Outputs: One unbounded queue carrying MarketRegimeChain values, consumed by a downstream broker.

Each regime observer is a pure lens carrier — it holds a single RegimeLens and no Reckoners, no prediction, no learning (see pg-exit-observer). It receives a MarketChain from each market observer, builds regime rhythm ASTs using regime_rhythm_specs(lens), composes lens-specific vocabulary as AST expressions, and produces a MarketRegimeChain that flattens all market fields plus adds regime_facts: Vec<ThoughtAST>. The broker is responsible for encoding; the regime observer only adds raw vocabulary and rhythms.

Per-stream default: M = 2 (one per regime lens: Core, Full).

Broker (broker_program)#

Role: Receive regime observations, submit papers to treasury, compose full thought, predict Hold/Exit via gate 4.

Inputs: One MarketRegimeChain queue from a regime observer. Outputs: Requests to the treasury via TreasuryHandle.

The broker composes the full thought (market rhythms + regime rhythms + portfolio rhythms + phase + time), encodes once, strips background via noise subspace, and predicts Hold/Exit via the gate reckoner. It submits papers to the treasury, reads outcome responses, and learns. See pg-broker for the full design.

Per-stream count: N × M = 6 × 2 = 12 brokers.

Treasury (treasury_program)#

Role: Manage capital, issue papers and real positions, resolve outcomes.

File: src/programs/app/treasury_program.rs (374 lines).

Inputs: A bounded queue of TreasuryEvent requests from brokers (via TreasuryHandle). Outputs: Per-broker response queues (1:1, not mailbox).

Brokers send requests via TreasuryHandle. The treasury processes them sequentially and responds. See pg-treasury for the full design.

Per-stream count: 1 treasury thread.

Data Flow Through the Grid#

flowchart TD
    main["Main thread\nsend ObsInput"] --> mkt_obs["Market Observer i\n(thread)\nrhythm → encode → predict"]
    mkt_obs -->|"Topic fan-out\nsend MarketChain"| reg0["Regime Observer 0"]
    mkt_obs --> reg1["Regime Observer 1"]
    mkt_obs --> regM["Regime Observer M"]
    reg0 -->|"send MarketRegimeChain"| broker["Broker [i, ei]\n(thread)\ncompose → encode → gate"]
    reg1 --> broker
    regM --> broker
    broker -->|"TreasuryHandle\nrequest/response"| treasury["Treasury\n(thread)"]

Each arrow is a Queue. The fan-out from market to regime observers uses Topic (1→N). Data flows forward only — brokers communicate with the treasury via TreasuryHandle (request/response queues).

Chain Types — Values on the Wire#

Defined in src/programs/chain.rs (38 lines). The module doc: "Each stage produces a new value. The type IS the proof of which stage produced it. No methods. No behavior. Values on the wire."

MarketChain (Clone)#

What the market observer produces. What the regime observer receives.

Field Type Meaning
candle Candle The enriched candle that triggered this step
window Arc<Vec<Candle>> Window of recent candles
encode_count usize Global monotonic encode counter
market_raw Vector Raw encoding of market features
market_anomaly Vector Anomaly component (deviation from noise subspace)
market_ast ThoughtAST The abstract syntax tree that was encoded
prediction Prediction Direction prediction (Up/Down)
edge f64 Edge/confidence score

MarketRegimeChain (not Clone)#

What the regime observer produces. What the broker receives. Flattens all MarketChain fields (duplicated, not borrowed) plus one regime-specific field:

Additional Field Type Meaning
regime_facts Vec<ThoughtAST> Regime-specific vocabulary facts and rhythm ASTs (no vectors, no distances)

The module doc: "No encoding. No vectors. No distances. The broker encodes."

Resource Pools and Orphan Detection#

Console, cache, and database handles are pre-allocated into HandlePools. Each program thread claims exactly one of each. Pools assert finish() — if any handle remains unclaimed, the program panics (T0-017, §"Resources allocated from pools").

Thread Count#

For one stream with N market observers, M regime observers, and one treasury thread (T0-017, §Thread count):

N + M + (N×M) + 1 (treasury) + 3 threads minimum (database + cache + console drivers).

With defaults (N=6, M=2): 6 + 2 + 12 + 1 + 3 = 24 threads.

Shutdown by Disconnection#

Programs do not use explicit shutdown flags. Shutdown cascades through Queue disconnection:

  1. Main thread drops candle_txs → market observers see Disconnected on input queues.
  2. Market observer threads are joined (collected back with trained state).
  3. TopicHandles are dropped → topics drain, then output senders drop → regime observers see Disconnected.
  4. Regime observer threads are joined.
  5. Broker input queues disconnect automatically → broker threads are joined.
  6. Finally, cache and database drivers are joined (T0-017, §Shutdown sequence).
