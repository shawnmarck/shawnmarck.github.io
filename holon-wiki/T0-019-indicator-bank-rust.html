T0-019: indicator_bank.rs — Verified Documentation WHAT

TL;DR

IndicatorBank is a stateful, streaming technical indicator engine. One instance per asset pair ("post"). Each call to tick(&Ohlcv) → Candle advances all internal state machines by one raw candle and returns an enriched Candle struct carrying ~65 computed indicator fields. These enriched candles are the sole input to the encoding pipeline (ThoughtEncoder). The file is self-contained at 2363 lines: streaming primitives, helper functions, the composed bank, and ~400 lines of tests.

Architecture Overview

Ohlcv (raw) ──► IndicatorBank.tick() ──► Candle (enriched)
                                            │
                                            ▼
                                     ThoughtEncoder
                                     (encoding pipeline)

One IndicatorBank per post (per asset pair). Stateful. Not thread-safe — accessed sequentially.

Upstream: Ohlcv from types::ohlcv (raw OHLCV + asset pair + timestamp). Downstream: Candle from types::candle (raw fields preserved + ~65 computed indicator f64s + phase label fields).

Phase 1: Advance all streaming primitives (lines 1641–1667)

27 step functions called sequentially, each pushing one candle's worth of data into its indicator(s):

step_sma → step_bollinger → step_rsi → step_macd → step_dmi → step_atr → step_stoch → step_cci → step_mfi → step_obv → step_volume_sma → step_roc → step_range_pos → step_trend_consistency → step_timeframe → step_ichimoku → step_persistence → step_vwap → step_kama_er → step_choppiness → step_dfa → step_var_ratio → step_entropy → step_aroon → step_fractal → step_divergence → step_price_action

Phase 2: Compute derived values (lines 1669–1866)

All indicator values are read from state and derived indicators are computed:

  • Bollinger Bands: upper/lower = sma20 ± 2σ, width = range/close, position = (close - lower) / range
  • Keltner Channels: upper/lower = ema20 ± 2×ATR, position similar
  • Squeeze: bb_width / kelt_width (continuous, not boolean)
  • ROC: rate of change at lags 1, 3, 6, 12
  • Range position: at 12/24/48 bar windows
  • Multi-timeframe: 1h/4h returns and body ratios (synthesized from 5m candles via ring buffers)
  • Ichimoku: tenkan, kijun, cloud_top, cloud_bottom, TK cross delta
  • Persistence: Hurst exponent, lag-1 autocorrelation (from 48-bar close buffer)
  • Regime features: KAMA efficiency ratio, choppiness index, DFA alpha, variance ratio, entropy rate, Aroon up/down, fractal dimension
  • Divergence: bull/bear RSI-price divergence detection
  • Stochastic cross delta
  • Price action: range ratio, gap, consecutive up/down counts
  • Timeframe agreement: sign agreement across 5m/1h/4h
  • Time features: minute, hour, day-of-week, day-of-month, month-of-year

Memory optimization: A single scratch: Vec<f64> (capacity 48) is taken via std::mem::take at the start of derived-value computation and returned at the end. This avoids split-borrow issues and prevents per-tick allocations.

Phase labeler (lines 1856–1866): After indicators, PhaseState.step() is called with smoothing = 2×ATR (Proposal 052). Phase history is only re-cloned when generation changes (~every 6 candles).

Phase 3: Update prev-state (lines 1868–1872)

Stores prev_tk_spread, prev_stoch_kd, prev_close, increments count.

Phase 4: Assemble enriched Candle (lines 1874–1948)

All computed values are packed into a Candle struct (65+ fields). Raw OHLCV fields are preserved from input.

Connection to Encoding System

In src/bin/wat-vm.rs line 704:

let candle = pipeline.bank.tick(&ohlcv);

The enriched Candle is then wrapped in Arc<Vec<Candle>> as a window and sent to observers. Observers extract indicator values as "facts" (via ThoughtAST), which are encoded by the ThoughtEncoder into sparse vectors for cosine similarity matching.

The encoding pipeline does NOT import IndicatorBank — it only consumes Candle. The bank is the sole producer of enriched candles.

Observations & Uncertainties

  1. VWAP never resets. vwap_cum_vol and vwap_cum_pv grow without bound. Over 652k candles this could produce very large floats. The comment doesn't address whether this is intentional (session VWAP) or an oversight.

  2. Bollinger stddev and SMA20 are independent. bb_stddev (RollingStddev period 20) and sma20 (SmaState period 20) both track close prices but don't share state. Bollinger width is 4σ / close. This is correct but means two separate rolling windows for the same data.

  3. Multi-timeframe synthesis is naive. 1h = 12 candles, 4h = 48 candles of 5m data. The "body" and "return" are computed over the oldest-to-newest span of the ring buffer. This approximates but doesn't aggregate (no proper OHLC rolling). The code acknowledges this implicitly by using compute_tf_body which treats oldest as open and newest as close.

  4. step_timeframe pushes the same H/L to both 1h and 4h buffers. Every 5m candle's high and low go into both buffers. This means 4h high is max of last 48 individual candle highs (not max of 48 aggregated 5m candles), which is actually correct for highest-high.

  5. Entropy bin thresholds are hardcoded: -0.5%, -0.1%, +0.1%, +0.5%. These are appropriate for 5m BTC but won't transfer to other assets without adjustment.

  6. prev_tf_1h_ret and prev_tf_4h_ret are stored but never read within tick(). They appear to be vestigial fields — the timeframe agreement computation uses compute_tf_ret directly rather than stored previous values.
