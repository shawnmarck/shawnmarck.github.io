T-1-003: The Guide (web version) — The Published Blueprint WHY

Source: algebraic-intelligence.dev/src/content/docs/blog/guide.md (2141 lines, ~103KB) Verified by: Subagent read — full document consumed Date: 2026-04-14

#What It Is

The 007 blueprint. A complete technical specification for the holon trading machine, written as a web-published blog post. It declares every struct, interface, dependency, and construction order. No implementation — only declarations. The document is structured as a walkable path: leaves first, root last, with every dependency satisfied before its consumer is introduced.

The web version has frontmatter (title, description) for the Astro-based website. It is intended to be read top-to-bottom by a newcomer who knows nothing about the project.

#Document Structure

#Frontmatter

title: "The Guide"

#Section 1: Holon-rs Primitives

The six substrate primitives provided by holon-rs (atom, bind, bundle, cosine, reckoner, curve) plus OnlineSubspace, ScalarEncoder, VectorManager, and utility operations. These are NOT specified in this tree — they are imported.

#Section 2: Definitions

Shared terminology and concepts:

  • Labels and measurement — Outcome (Grace/Violence), Prediction (Discrete/Continuous), TradePhase, MarketLens, ExitLens
  • Data and encoding — Candle fields, ATR, RingBuffers, vocabulary domains
  • Observers and learning — Noise subspace, conviction, the broker, proof curve, the journal
  • Trade lifecycle — Paper trades, settlements, residue, the runner
  • Infrastructure — Window sampler, ctx (immutable shared context), the ledger

#Section 3: Forward Declarations

The path from market to thought (ASCII diagram showing raw-candle → candle → indicators → vocabulary → encoder → thought → observer → broker → proposal). The construction order listing every entity in build sequence.

  12. Post

Each entity declares: type, fields with types, interfaces (functions it provides), dependencies on prior entities, and key design notes.

#Section 5: The Build Order

Summary construction order — reiterates leaves-to-root.

#Section 6: The CSP Per Candle

The Communicating Sequential Process model — four steps per candle:

  1. Produce (parallel) — posts observe candles, encode thoughts, brokers propose
  2. Fund (sequential) — treasury evaluates proposals, funds proven ones
  3. Propagate (sequential) — settlements route outcomes back to observers
  4. Collect (sequential) — process logs, drain queues, prepare next candle

Channel types and typed boundaries between steps.

ASCII art dataflow diagram showing the entire system's data flow.

#Key Declarations

#The reckoner interface

(make-reckoner config) → Reckoner   ; config = (dims, recalib-interval, mode)
(observe reckoner thought observation weight)
(predict reckoner thought) → Prediction
(decay reckoner factor)
(experience reckoner) → f64

Two modes: Discrete (labels: Up/Down classification) and Continuous (default-value: scalar regression).

#The curve (as standalone)

(make-curve) → Curve
(record-prediction curve conviction correct?)
(edge-at curve conviction) → f64
(proven? curve min-samples) → bool

Note: The web guide has the curve as a standalone primitive with (make-curve). The lab guide changed this — the curve became internal to the reckoner.

#The enterprise expression

(on-candle enterprise raw-candle ctx)

This is THE function — the single entry point that runs the entire four-step loop.

#Trade phases (web version)

  • Active → Settled(Violence) on safety-stop
  • Active → PrincipalRecovered on take-profit
  • PrincipalRecovered → Runner (wider trailing stop, zero cost basis)
  • Runner → RunnerSettled(Grace) on runner-trail stop

Note: The lab guide significantly changed the trade lifecycle. See comparison below.

#Distances (web version)

4 learnable scalars: trail, stop, tp, runner-trail

Note: The lab guide reduced this to 2: trail and stop.

#Exit observers (web version)

4 continuous reckoners (one per distance: trail, stop, tp, runner-trail)

Note: The lab guide reduced this to 2.

#N and M (web version)

N = 6 market observers, M = 4 exit observers → N×M = 24 brokers

Note: The lab guide removed the specific numbers.

#Key Technical Divergences

  1. The Curve — Standalone vs. Internal

    • Web: (make-curve) → standalone Curve object
    • Lab: Curve is internal to the Reckoner. (resolve reckoner conviction correct?) and (edge-at reckoner conviction) — no standalone curve.
    • Assessment: Lab is the evolved design. The book narrative confirms the curve dissolved into the reckoner: "one scalar fact in the bundle."

  2. Trade Lifecycle — Simplification

    • Web: 4 phases with separate runner phase (Active → PrincipalRecovered → Runner → RunnerSettled)
    • Lab: Runner is a stop-management phase within a single active position. One exit. One swap. Grace/Violence determined by whether exit amount exceeds principal.
    • Assessment: Lab is significantly more detailed (~100 lines of concrete examples including number flow for USDC→WBTC trades). The web version's PrincipalRecovered and RunnerSettled were collapsed.

  3. Distances — 4 vs. 2

    • Web: 4 distances (trail, stop, tp, runner-trail) with 4 exit reckoners
    • Lab: 2 distances (trail, stop) with 2 exit reckoners
    • Assessment: Lab simplified. The take-profit and runner-trail distances were removed. The runner concept was redesigned as a phase, not a separate distance.

  4. Reckoner Constructor Signature

    • Web: (make-reckoner config) — config contains dims, recalib-interval, mode
    • Lab: (make-reckoner name dims recalib-interval config) — name, dims, recalib-interval separated from config; config specifies readout mode only
    • Assessment: Lab is more explicit — separates identity and configuration from mode specification.

  5. Magic Numbers — 4 vs. 2

    • Web: k_trail, k_stop, k_tp, k_runner_trail
    • Lab: k_trail, k_stop (only)
    • Assessment: Consistent with the distance simplification.

  6. New Entities in Lab (not in Web)

    • EngramGate — depends on Outcome enum, Reckoner, OnlineSubspace
    • Simulation — depends on Distances
    • Performance section — performance characteristics documentation
    • Forge coordinates — reference to the forge ward
    • Assessment: These are additions discovered during the 32-pass ignorant reader process.

  7. Stop Breathing

    • Lab adds significant detail: "The stops breathe" — every candle, step 3c re-queries the exit observer for updated distances. Stops adapt continuously, not set-and-forget. Includes on-chain update cost consideration.
    • Assessment: Lab is much more operationally detailed.

  8. Concrete Number Flow Examples

    • Lab adds two detailed worked examples (USDC→WBTC long trade, WBTC→USDC short trade) with exact dollar amounts at each stage.
    • Web has no such examples.
    • Assessment: Lab is significantly more tutorial in this section.

  9. The amplify utility signature

    • Web: (amplify vec vec weight) — has a duplicated vec parameter
    • Lab: (amplify vec weight) — corrected
    • Assessment: Lab fixed a bug found by the ignorant ward.

  10. N and M — Specific vs. General

    • Web: "N is the number of market observers (today: 6, one per MarketLens variant). M is the number of exit observers (today: 4)"
    • Lab: "N is the number of market observers (one per MarketLens variant). M is the number of exit observers (one per ExitLens variant). N and M are determined by the lens enums — add a variant, add an observer."
    • Assessment: Lab is more general — removed hardcoded counts, made the system self-describing.

#Summary of Divergence

The web guide is a snapshot published before the full ignorant reader process completed. The lab guide is the evolved specification after 32 passes of the ignorant ward, designer consultations, and architectural refinements. The core structure is identical; the differences are in:

  1. The curve — moved from standalone to reckoner-internal
  2. The trade lifecycle — simplified from 4 phases to 2 with runner as phase-not-event
  3. The distances — reduced from 4 to 2 (tp and runner-trail removed)
  4. New entities — EngramGate, Simulation added
  5. Operational detail — much more concrete in lab (stop breathing, number flow, update costs)
  6. Bug fixes — amplify signature, forward reference ordering

Verdict: The lab guide is the authoritative version. The web guide is a published artifact that predates several significant architectural decisions. Anyone implementing the system should use the lab guide as the source of truth.
