Encoding Cache — the distributed LRU encoding service EXPLAIN

#TL;DR

The Encoding Cache (<a class="wikilink" href="pg-encoding-cache.html">EncoderService</a>) is a single-threaded pipe loop that owns the persistent LRU composition cache for <a class="wikilink" href="pg-thought-system.html">ThoughtAST</a>→<a class="wikilink" href="T0-001-vector.html">Vector</a> mappings. It serves N concurrent callers without any mutex — each caller gets its own three-pipe EncoderHandle, and the service thread iterates all handles once per loop iteration. Callers compute vectors locally on cache misses and fire-and-forget install them back. The cache is purely a store — it never computes, only stores and retrieves. Three Arc<AtomicUsize> counters (hits, misses, cache-size) provide lock-free observability. Shutdown is automatic when all handles drop.

#What the encoding cache IS

The <a class="wikilink" href="pg-thought-encoding.html">ThoughtEncoder</a> produces vectors from ASTs, but recomputing every subtree on every candle is expensive. The encoding cache is the solution: a shared, cross-candle LRU cache that sits between callers and the encoder. It stores the results of expensive tree walks so that structurally identical ASTs can skip re-evaluation entirely.

"ThoughtEncoder cache as a single-threaded pipe loop. The encoder holds an LRU cache. Each caller gets its own pipe set (EncoderHandle). The loop iterates all pipes once per iteration. No mutex. One thread. N callers. One cache." — encoder-service.wat, lines 1–7 (T2-002)

The cache is the persistent layer; the <a class="wikilink" href="pg-thought-encoding.html">ThoughtEncoder</a>'s own composition map serves as a per-candle working set.

"The ThoughtEncoder holds a local per-candle working set; the EncoderService holds the persistent cross-candle LRU cache shared across all N observer threads." — T2-002, Relationship to ThoughtEncoder

#Architecture — one thread, N callers, zero mutexes

#EncoderHandle — per-caller pipe set

Each caller receives an EncoderHandle containing three pipes:

(struct encoder-handle
  [lookup  : Sender<ThoughtAST>]          ; bounded 1
  [answer  : Receiver<Option<Vector>>]    ; bounded 1
  [install : Sender<(ThoughtAST, Vector)>]) ; unbounded, fire and forget


(T2-002, lines 22–25)






  Pipe
  Capacity
  Direction
  Purpose






  lookup
  bounded 1
  caller → service
  Send a <a class="wikilink" href="pg-thought-system.html">ThoughtAST</a> for cache lookup




  answer
  bounded 1
  service → caller
  Receive Some(Vector) (hit) or None (miss)




  install
  unbounded
  caller → service
  Fire-and-forget insertion of computed (AST, Vector) pair






#encoder-get — blocking lookup


Sends AST on lookup, blocks on answer, receives Option<Vector>. Bounded-1 on both sides provides natural backpressure.


(T2-002, lines 28–31)


#encoder-set — fire-and-forget install


Sends (AST, Vector) pair on install. Non-blocking, unbounded capacity. The caller does not wait for confirmation.


(T2-002, lines 34–36)




#Service thread loop — three phases per iteration


The service runs an infinite loop. Each iteration has three phases:


#Pass 1 — Drain installs


Iterates all install receivers. Drains each one completely via try-recv. Every (ast, vec) pair is inserted into the LRU cache. Updates the cache-size atomic.




"Pass 1: drain ALL install pipes. Put into cache."
— encoder-service.wat, line 84 (T2-002)




Critical ordering guarantee: installs are drained before lookups are serviced. This means a value installed on the previous loop iteration is available for lookup on the current one, implementing eventual consistency with a single-iteration latency: "miss on candle N, hit on N+1".




"The GUIDE (line 1440) declares the compositions field as LruCache<ThoughtAST, Vector> and explains the eventual-consistency model (lines 1441–1446): 'miss on candle N, hit on N+1'. The EncoderService implements this."
— T2-002, Relationship to ThoughtEncoder




#Pass 2 — Service lookups


Iterates all lookup receivers. For each pending AST: checks the LRU cache, increments hits or misses atomic accordingly, sends result (Some(Vector) or None) back on the corresponding answer pipe. On channel disconnect: marks that handle as closed.




"Pass 2: service ALL pending lookup pipes."
— encoder-service.wat, line 93 (T2-002)




#Pass 3 — Shutdown check & idle blocking


If all handles are disconnected, the loop breaks and the thread exits. Otherwise, uses select over all open pipes to block until any channel has data. Zero CPU when idle.




"Block until ANY channel has data. Zero CPU when idle."
— encoder-service.wat, lines 112–113 (T2-002)






#Protocol summary


flowchart TD
    caller_lookup["Caller"] -->|"write AST to lookup pipe"| block["block on answer"]
    block -->|"receive Some/None"| result{"Some\n(hit) or None\n(miss)"}
    result -->|"None"| compute["compute locally"]
    compute -->|"write (AST, Vector)\nto install pipe\n(fire and forget)"| drain

    subgraph encoder["Encoder service loop"]
        drain["Pass 1: drain installs"] --> service["Pass 2: service lookups"]
        service --> select["Pass 3: select,\nblock until data"]
        select --> drain
    end



(T2-002, lines 9–12)


This implements the "values up" principle: callers compute locally on cache misses rather than requesting computation from the service. The service is purely a cache — it never computes vectors, only stores and retrieves them.




"The encode function NEVER writes to the cache. Misses are returned as values — the enterprise collects them and inserts between candles. Values up, not queues down."
— thought-encoder.wat, lines 11–13 (T2-001)






#EncoderService struct


(struct encoder-service
  [handle     : JoinHandle<()>]
  [hits       : Arc<AtomicUsize>]         ; cache hit counter
  [misses     : Arc<AtomicUsize>]         ; cache miss counter
  [cache-size : Arc<AtomicUsize>])        ; current cache occupancy



(T2-002, lines 43–47)


The service does not hold sender copies — the handles ARE the senders. When all handles drop, the channels close and the encoder thread exits automatically.




"Does NOT hold sender copies — the handles ARE the senders. When all handles drop, the channels close, the cascade flows, the encoder thread exits."
— encoder-service.wat, lines 39–41 (T2-002)

#Spawn and shutdown


#encoder-service-spawn(n-callers, cache-capacity) → (EncoderService, Vec<EncoderHandle>)


Creates the service and N handles. For each caller, creates three pipe pairs (lookup/answer bounded-1, install unbounded). Spawns the service thread with the receiver halves.


(T2-002, lines 51–122)


#encoder-service-shutdown(svc)


Joins the thread handle. Requires that all handles have already been dropped.




"The cascade must have already closed all handles."
— encoder-service.wat, line 125 (T2-002)






#Relationship to ThoughtEncoder memory model


The <a class="wikilink" href="pg-thought-encoding.html">ThoughtEncoder</a> maintains two kinds of memory with fundamentally different semantics:






  Memory
  Semantics
  Owner
  Eviction






  Atoms — dictionary
  Finite, pre-computed, permanent
  <a class="wikilink" href="T0-005-vector-manager.html">VectorManager</a>
  Never




  Compositions — cache
  Infinite, optimistic, use-if-we-have-it
  EncoderService (LRU)
  Capacity-based








"Finite. Known at startup. Pre-computed. Never evicted."
— thought-encoder.wat, line 8 (T2-001)






"Infinite. Optimistic. Use it if we have it. Compute if we don't."
— thought-encoder.wat, line 9 (T2-001)




The GUIDE declares compositions as LruCache<ThoughtAST, Vector> (GUIDE line 1440). The wat file declares it as Map<ThoughtAST, Vector> with capacity 4096, annotated "LRU moved to EncoderService". Per CLAUDE.md: "When layers diverge, the guide is right." The EncoderService holds the true LRU; the <a class="wikilink" href="pg-thought-encoding.html">ThoughtEncoder</a>'s composition map is the per-candle working set.




"Same structure, same vector."
— GUIDE.md, line 1475 (T2-001)




The AST structure IS the cache key — structural equality (with round_to quantization at vocabulary emission time for reliable float hashing) determines identity.




#Design principles






  Principle
  How the cache implements it






  No mutex
  Single-threaded ownership of the cache; callers communicate via pipes




  Values up, not queues down
  Callers compute on miss; the cache never computes




  Eventual consistency
  Install-drain-before-lookup ordering gives one-iteration latency




  Graceful shutdown
  Handle-drop cascades channel closes; thread auto-exits




  Zero-idle CPU
  select blocks until any pipe has data




  Lock-free stats
  Arc<AtomicUsize> for hit/miss/size — readable from any thread
