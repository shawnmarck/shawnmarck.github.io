Encoding Cache — generic batch-only LRU cache program EXPLAIN

TL;DR#

The encoding pipeline produces vectors from ASTs. The AST-to-vector algebra lives in encoding::encode::encode() with a two-tier cache (L1 per-entity + L2 shared). The L2 cache is a generic cache::<K, V>() program — a single-threaded driver that owns a persistent LRU store behind crossbeam pipe pairs. It serves N concurrent callers without any mutex — each caller gets its own CacheHandle<K, V> (two bounded-1 pipes: request + response). The driver blocks on crossbeam Select, drains all pending requests, groups them writes-first/reads-second, processes in batch, and responds. Atomic counters track hits and misses. Telemetry flows through a gate pattern (CacheStats). Shutdown is automatic when all handles drop.

What the encoding cache IS#

The encoding pipeline produces vectors from ASTs via the free function encode(state, l2_cache, ast, vm, scalar), but recomputing every subtree on every candle is expensive. The encoding cache (L2) is the sharing layer. L1 is the memo: a per-entity LruCache<u64, Vector> inside EncodeState that absorbs repeated lookups across candles within a single entity. The L2 cache is a shared, cross-entity cache program (cache::<K, V>()) that sits behind pipes — callers batch-get/batch-set through CacheHandle<K, V>.

The encode() function uses both tiers in four phases:

  1. Walk the tree, collect L1 misses (by precomputed u64 key).
  2. Batch-get L2 for L1 misses. Populate L1 from L2 hits.
  3. Compute remaining misses sequentially with L1 memoization.
  4. Install computed results into L1 and confirmed-batch-set to L2.

Architecture — one thread, N callers, zero mutexes#

CacheHandle<K, V> — per-caller pipe pair#

Each caller receives a CacheHandle<K, V> containing two bounded-1 pipes:

(struct cache-handle
  [req-tx  : Sender<CacheRequest<K, V>>]   ; bounded 1, caller → driver
  [resp-rx : Receiver<CacheResponse<K, V>>]) ; bounded 1, driver → caller

pub struct CacheHandle<K, V> {
    client_idx: usize,
    req_tx: QueueSender<CacheRequest<K, V>>,
    resp_rx: QueueReceiver<CacheResponse<K, V>>,
}
Pipe Capacity Direction Purpose
req_tx bounded 1 caller → driver Send BatchGet or BatchSet request
resp_rx bounded 1 driver → caller Receive BatchGet(results) or BatchSetAck

Request / response types#

enum CacheRequest<K, V> {
    BatchGet { client: usize, keys: Vec<K> },
    BatchSet { client: usize, entries: Vec<(K, V)> },
}

enum CacheResponse<K, V> {
    BatchGet(Vec<(K, Option<V>)>),   // each key paired with hit/miss
    BatchSetAck,                      // confirmed install
}

batch_get — blocking batch lookup#

Sends BatchGet { keys }, blocks on response, receives Vec<(K, Option<V>)> — each key paired with its lookup result. Returns None if the driver has disconnected.

pub fn batch_get(&self, keys: Vec<K>) -> Option<Vec<(K, Option<V>)>>

batch_set — confirmed batch install#

Sends BatchSet { entries }, blocks until the driver acknowledges. The driver has committed all entries to the LRU before the ack is sent. No fire-and-forget — every install is confirmed.

pub fn batch_set(&self, entries: Vec<(K, V)>)

Driver loop — writes-first, reads-second#

The driver runs an infinite loop inside cache::<K, V>():

Block on Select#

Uses crossbeam::channel::Select to block until any client request queue has data. Zero CPU when idle. If all clients have disconnected, the driver emits final telemetry and exits.

Drain all pending requests#

Iterates all client queues, draining every pending try_recv. Requests are grouped into two bins: writes (BatchSet) and reads (BatchGet). Disconnected clients are marked closed.

Process writes first#

All BatchSet requests are processed before any reads. Each entry is inserted into the LRU. BatchSetAck is sent to the caller. The writes-first ordering ensures freshest data is visible to subsequent reads in the same iteration.

Process reads second#

All BatchGet requests are processed next. Each key is looked up in the LRU. Results are paired as (K, Option<V>) and sent back. The driver uses batched atomic counter updates — one fetch_add per batch (not per key) for both hits and misses.

Gate telemetry#

After each iteration, the driver checks can_emit(). If the gate is open, it snapshots CacheStats (hits, misses, cache_size, ns_gets, ns_sets, gets_serviced, sets_drained, evictions), calls emit(stats), and resets the accumulator. This avoids per-key overhead for telemetry — one emission per gate open.

This implements the two-tier values-up principle: callers compute on L1 miss; computed results are confirmed-batch-set to L2 for cross-entity sharing. The cache never computes — it stores and retrieves.

CacheDriverHandle and CacheStats#

CacheDriverHandle#

pub struct CacheDriverHandle {
    name: String,
    thread: Option<thread::JoinHandle<()>>,
    pub hits: Arc<AtomicUsize>,    // total cache hits since startup
    pub misses: Arc<AtomicUsize>,  // total cache misses since startup
}

No Drop impl — joining in Drop would deadlock if senders are still alive. The shutdown cascade guarantees correctness: all CacheHandle drops → driver drains remaining requests → driver exits. Call join() explicitly when you need to wait.

CacheStats#

pub struct CacheStats {
    pub hits: usize,
    pub misses: usize,
    pub cache_size: usize,
    pub ns_gets: u64,
    pub ns_sets: u64,
    pub gets_serviced: usize,
    pub sets_drained: usize,
    pub evictions: usize,
}

Emitted through the gate pattern: can_emit() / emit(stats) callbacks passed to cache(). The gate avoids per-key overhead — stats accumulate between gates, one snapshot per open.

Creation and shutdown#

cache::<K, V>(name, capacity, num_clients, can_emit, emit) → (Vec<CacheHandle<K, V>>, CacheDriverHandle)#

Creates N CacheHandles (one per caller) and the driver thread. For each client, creates two bounded-1 pipe pairs (request + response). Spawns the driver with the receiver halves. The driver owns the LruCache<K, V> and polls client queues directly — no mailbox, no fan-in thread.

pub fn cache<K, V>(
    name: &str,
    capacity: usize,
    num_clients: usize,
    can_emit: Box<dyn Fn() -> bool + Send>,
    emit: Box<dyn Fn(CacheStats) + Send>,
) -> (Vec<CacheHandle<K, V>>, CacheDriverHandle)
where
    K: Send + Clone + Hash + Eq + 'static,
    V: Send + Clone + 'static,

CacheDriverHandle::join(self)#

Blocks until the driver thread exits. The driver exits when all client handles have been dropped. The cascade is the shutdown guarantee.

Two-tier memory model#

The encoding pipeline uses two cache tiers with fundamentally different scopes:

Tier Semantics Owner Eviction Keys
L1 — per-entity memo Local LruCache<u64, Vector> in EncodeState Per observer thread Capacity-based (default 16 384) Precomputed u64 hashes
L2 — shared cache program cache::<u64, Vector>() behind pipes Single driver thread Capacity-based Precomputed u64 hashes

Key design#

  • Precomputed hashes: ThoughtAST::precomputed_hash() produces a u64 at AST construction time. All cache lookups use integer keys — O(1), nanoseconds, no tree walking.
  • L1 is the memo: absorbs repeated lookups across candles within one entity. No pipe overhead — direct LruCache access.
  • L2 is the sharing layer: cross-entity sharing via the generic cache() program. Observers in different entities can benefit from each other's computations.

Atoms (never evicted)#

The permanent atom dictionary lives in VectorManager — finite, pre-computed, never evicted. Atoms are resolved by name, not cached.

Design principles#

Principle How the cache implements it
No mutex Single-threaded driver owns the LRU; callers communicate via crossbeam bounded-1 pipe pairs
Values up, not queues down Callers compute on L1 miss; results confirmed-batch-set to L2
Two-tier L1 per-entity memo + L2 shared cache program
Precomputed hashes u64 keys, O(1) lookups, no tree walking
Writes-before-reads BatchSet processed before BatchGet in each iteration — freshest data for reads
Graceful shutdown Handle-drop cascades channel disconnects; driver auto-exits
Zero-idle CPU crossbeam Select blocks until any client queue has data
Batched atomics One fetch_add per batch, not per key
Gate telemetry CacheStats emitted through can_emit/emit callbacks — no per-key overhead
