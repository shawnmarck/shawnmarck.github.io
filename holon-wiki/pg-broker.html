Broker — the accountability and execution primitive EXPLAIN

TL;DR: The broker binds exactly one market observer to one position observer, forming a cell in an N×M grid. It owns paper trades, scalar accumulators for trail/stop distances, and a dollar-P&L gate. Every output flows upward as return values — the broker never mutates observer state directly. It is the unit of accountability: it proposes, tracks, resolves, and decides whether it has earned the right to trade.

What it is#

"The broker IS the accountability unit. It owns paper trades, scalar accumulators, and a Grace/Violence reckoner. Values up, not effects down." — broker.rs

The broker sits at the intersection of a market observer (which predicts direction) and a position observer (which sizes risk). It translates their composed thought into a paper trade, watches that trade against live prices, and propagates outcomes back to both observers via PropagationFacts. Crucially:

  • The broker proposes but does not execute. Papers are hypothetical.
  • The broker learns but does not mutate. All learning returns as PropagationFacts; the Post routes them.
  • The broker gates itself. A broker with negative expected value cannot propose funded trades.

The N×M Grid#

Brokers live in a flat array indexed by slot_idx. With N market observers and M position observers:

slot_idx = market_idx × M + position_idx
market_idx = slot_idx / M       (integer division)
position_idx = slot_idx % M


Each broker's slot_idx is its identity — it determines which market observer and which position observer it binds. For example: slot_idx=5, M=3 → market_idx=1, position_idx=2. (T0-018, lines 217–224)

The Gate#


pub fn gate_open(&self) -> bool {
    let cold_start = self.grace_count < 50 || self.violence_count < 50;
    cold_start || self.expected_value > 0.0
}



Two phases:


    

  1. Cold start — if either outcome count is below 50, the gate is always open. Not enough data to judge.
    2. 

  2. Warm — after ≥50 of both Grace and Violence outcomes, the gate opens only when expected_value > 0.0. A broker that costs money on average is suppressed.
    3. 



The expected value formula:


expected_value = grace_rate × avg_grace_net + (1 − grace_rate) × avg_violence_net



Where avg_grace_net and avg_violence_net are EMAs (α=0.038, half-life ~50 papers) of dollar-P&L per paper against a $10,000 reference capital, including swap fees on both entry and exit. (T0-018, §7.8)




The Distance Cascade#


When a broker needs trail/stop distances, it consults a three-tier fallback:






  Tier
  Source
  Condition






  1
  Reckoner answer (from the position observer's continuous Reckoner)
  Some(dists) → use immediately




  2
  Scalar accumulators
  If the accumulator has observations, extract with 100 samples, clamped to [0.1%, 10%]




  3
  default_distances
  Constructor-provided default if no learned data exists






The broker owns tiers 2 and 3. Tier 1 comes from outside. (T0-018, §7.4)




Paper Trades#


Each candle, the broker calls register_paper() to create a new PaperEntry. The paper:


    

  • Records the composed thought (market + position), the raw market thought, and the position thought separately (Proposal 026: position learns from its own thought, not composed)
    • 

  • Tracks extreme (best price in predicted direction) and trail_level (trailing stop)
    • 

  • Has two triggers: trail (Grace — prediction confirmed) and stop (Violence — prediction wrong)
    • 

  • Accumulates price history for deferred reckoner training
    • 



The paper lifecycle (T0-015):


flowchart TD
    created["Created"] -->|"Stop fires before trail crosses"| violence["Violence"]
    created -->|"Trail crosses"| signaled["signaled (now a Runner)"]
    signaled -->|"Reversal crosses trailing stop"| grace_resolution["Grace resolution"]
    signaled -->|"Continues favorably"| keeps_running["keeps running"]
    created -->|"Neither fires"| stays_open["stays open"]



Key ordering: Violence is checked before Grace. If both would fire on the same tick, Violence wins. (T0-015, §tick)




Runners and Deferred Training#


When a paper signals Grace (trail crosses), it becomes a runner. While running, the broker accumulates a RunnerHistory — parallel arrays of composed thoughts, distances, and prices at each candle.


When the runner resolves (trailing stop fires), compute_position_batch() produces deferred training data for the position observer:


    

  1. Compute optimal stop via full simulation over the runner's price history
    2. 

  2. Suffix extremum pass — O(n) backward scan for the best remaining price from each candle
    3. 

  3. Per-candle optimal trail from the suffix extremum
    4. 

  4. Sparse sampling — only emit a training sample when optimal trail changed by >10% relative, avoiding redundant samples
    5. 

  5. Weight = the optimal trail value (what that distance would capture)
    6. 



This batch is included in the Resolution struct as position_batch and routed to the position observer by the Post. (T0-018, §8.2)




Propagation and Learning#


propagate() is called by the Post with each resolved paper's data. It performs three steps:


    

  1. Track record — increment cumulative grace/violence and trade counts
    2. 

  2. Scalar accumulators learn — feed optimal (hindsight) trail/stop distances, weighted by outcome
    3. 

  3. Dollar P&L — compute net P&L including swap fees against $10,000 reference capital, update EMAs of avg_grace_net and avg_violence_net (α=0.038), recompute expected_value
    4. 



The method returns PropagationFacts — a data struct containing which observers should learn and with what data. The broker does not talk to observers directly. (T0-018, §7.8)

Direction Flip: close_all_runners()#


When the market observer's active direction flips, close_all_runners() force-resolves all active runners with Outcome::Grace. Non-runner papers that haven't resolved survive the flip and will hit their stop naturally. (T0-018, §7.6)

Verified Properties#


    

  1. No shared mutation. propagate() returns PropagationFacts. tick_papers() returns Vec<Resolution>. The broker never touches observer state. (T0-018, §12)
    2. 

  2. Runner history is scoped. Created at Grace signal, accumulated per-candle, consumed at runner resolution. Keyed by paper_id, preventing stale data. (T0-018, §12)
    3. 

  3. Gate has two distinct modes with a clear boundary at 50 resolved trades. (T0-018, §12)
    4. 

  4. Distance cascade has exactly three tiers with clean fallback. (T0-018, §12)
    5. 

  5. P&L includes swap fees on both entry and exit. (T0-018, §12)
    6. 

  6. 15 tests cover construction, grid math, gate logic, paper lifecycle, propagation, dollar P&L, EMA convergence, accumulator naming, and optimal distance approximation. (T0-018, §9)
    7.
