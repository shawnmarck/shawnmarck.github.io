2. fund-proposals (line 63) — Core funding logic

Evaluates all proposals, sorted by edge (descending). Funds those that pass two gates; rejects the rest. Drains proposals after processing.

Funding gates (checked per proposal):

  1. Edge gate: edge < venue-cost-rate → rejected ("edge below venue cost")
  2. Capital gate: available-capital <= 0.0 → rejected ("insufficient capital")

Venue cost rate: 2.0 × (swap-fee + slippage) — the round-trip cost (open + close).

Funding math:

  • reserve-amount = amount + (amount × venue-cost-rate) — the total capital locked
  • actual-reserve = min(reserve-amount, available-capital) — clamped to what's available
  • trade-amount = actual-reserve / (1.0 + venue-cost-rate) — back-computed principal
  • Capital moves: available → reserved by trade-amount
  • Entry price initialized to 0.0 (set later by enterprise from current price)
  • Stop levels computed via distances-to-levels(distances, entry-price, side)
  • Trade stored in trades map, origin stored in trade-origins map
  • Proposals vector drained (set to empty list) after processing

Returns: Vec — one per proposal (funded or rejected).

3. settle-triggered (line 133) — Settlement logic

Checks all active trades against their stop-levels using current prices.

Input: treasury, Map<(Asset, Asset), f64> (current prices)

Two trigger paths:

Path A — Safety stop fires (line 146):

  • Condition: trade is :active, price crosses safety-stop in the adverse direction
  • Buy: price ≤ safety-stop; Sell: price ≥ safety-stop
  • Exit value: amount × (1.0 - venue-cost-per-swap) — loses the venue cost on exit
  • Loss: amount - exit-value
  • Capital: exit-value returns to available; full amount removed from reserved
  • Outcome: :settled-violence, reason :violence

Path B — Trailing stop fires (line 172):

  • Condition: trade is :active or :runner, price crosses trail-stop in the adverse direction
  • Buy: price ≤ trail-stop; Sell: price ≥ trail-stop
  • Exit value: amount × (price / entry-price) × (1.0 - venue-cost-per-swap) — proportional P&L minus cost
  • Outcome: :grace if exit-value > amount (profit), :violence otherwise (loss)
  • Residue: exit-value - amount (positive = profit, negative = loss)
  • Capital: min(exit-value, amount) returns to available; full amount removed from reserved
  • Phase: :settled-grace or :settled-violence depending on outcome

Both paths: trade and origin removed from their maps.

Returns: (Vec, Vec) — parallel lists.

Bounded Loss — How It Works

The spec declares bounded loss through a capital reservation model:

  1. At funding: Capital is moved from available to reserved. The amount reserved equals the trade's principal (trade-amount). The reserve-amount includes a buffer for venue costs, but only the principal is actually moved.
  1. At settlement (safety stop): The maximum loss is amount × venue-cost-per-swap — the cost of one swap (exit only). The rest of the principal (amount × (1 - venue-cost-per-swap)) returns to available. This is the hard bound: the system cannot lose more than one venue-cost fraction of the trade principal.
  1. At settlement (trailing stop): The trade's P&L is proportional to price movement. If the trade wins, profit returns as available capital. If the trade loses beyond venue cost on exit, the loss is bounded by the same formula as the safety stop case.
  1. The bound is structural, not policy. The reserve map is the accounting enforcement: capital in reserved cannot be deployed. It only returns (to available) through settlement. No double-spending is possible because proposals are drained after funding.

Observations (stranger's eye)

  • total-equity comment says "converted to denomination" but the code sums raw f64 values without conversion. If multiple assets exist, this is a bug or an incomplete spec.
  • fund-proposals sorts by edge descending and greedily funds top-to-bottom. First proposal to deplete an asset's capital blocks subsequent proposals for that asset.
  • The runner transition is checked inside update-trade-stops, which is called from outside the treasury (by the post's tick step 3c).
  • Entry price is set to 0.0 at funding time; the comment says "set by the enterprise from current price." This means the enterprise must update the trade's entry-price between funding and the first tick. If it doesn't, stop-level computation in settle-triggered would divide by zero (price / entry-price).
  • Trade origins are stored purely for propagation at settlement time — composed-thought and prediction are extracted and embedded in the settlement.
