Chain Types EXPLAIN

Chain Types

The typed values that flow through the organism.

TL;DR — Two pure-data structs in src/programs/chain.rs (40 lines) carry all information between pipeline stages. The type name is the proof of which stage produced it. No methods, no behavior, no shared references — just values on the wire.

Design Principle

MarketChain — Clone

Transport: Bounded Queue (capacity 1) into market observer; Topic (1→N) out to position observers.

FieldTypeMeaning
candleCandleThe enriched OHLCV candle that triggered this step
windowArc>Window of recent candles (currently length 1)
encode_countusizeGlobal monotonic encode counter
market_rawVectorRaw encoding of market features via VectorManager
market_anomalyVectorAnomaly component (deviation from noise subspace)
market_astThoughtASTThe abstract syntax tree that was encoded
predictionPredictionDirection prediction (Up / Down)
edgef64Edge / confidence score

Implements Clone because a Topic clones the message to every subscriber. This is the only chain type that needs fan-out.

MarketPositionChain — not Clone

Transport: Bounded Queue (point-to-point, one broker per chain).

Flattens all MarketChain fields — duplicated, not borrowed — then adds position-specific fields:

Inherited Field (renamed)OriginalMeaning
candleMarketChain::candleSame enriched candle
windowMarketChain::windowSame window
encode_countMarketChain::encode_countSame counter
market_rawMarketChain::market_rawSame market raw encoding
market_anomalyMarketChain::market_anomalySame market anomaly
market_astMarketChain::market_astSame market AST
market_predictionMarketChain::predictionRenamed from prediction
market_edgeMarketChain::edgeRenamed from edge
New FieldTypeMeaning
position_rawVectorPosition observer's own raw encoding
position_anomalyVectorPosition observer's anomaly vector
position_astThoughtASTPosition observer's AST
position_distancesDistancesTrailing stop and safety stop distances from Reckoner

Data Flow Through the Grid

flowchart TD
    main["Main thread\nsend ObsInput"] --> mkt_obs["Market Observer i\n(thread)\nencode → predict"]
    mkt_obs -->|"Topic fan-out\nsend MarketChain"| pos0["Position Observer 0"]
    mkt_obs --> pos1["Position Observer 1"]
    mkt_obs --> posM["Position Observer M"]
    pos0 -->|"send MarketPositionChain"| broker["Broker [i, ei]\n(thread)"]
    pos1 --> broker
    posM --> broker
    broker -.->|"ObsLearn → market observer i's mailbox"| mkt_obs
    broker -.->|"PositionLearn → position observer ei's mailbox"| posM
    broker -.->|"TradeUpdate → position observer ei's trade mailbox"| posM

See signal-propagation for the full wiring table including ObsLearn, PositionLearn, and TradeUpdate return-path signals.

Key Design Choices

ChoiceRationale
Pure data, no methodsTypes carry information; behavior lives in the program functions that produce and consume them.
Flattened, not composedMarketPositionChain duplicates market fields rather than holding a MarketChain field. This avoids lifetime complexity and shared-reference entanglement.
MarketChain clones; MarketPositionChain does notClone is required only for fan-out to multiple position observers. Brokers receive point-to-point, so ownership transfer suffices.
Arc> for windowThe one shared allocation — read-only, never mutated after creation.
Type-is-proofThe compiler ensures a MarketPositionChain can only be produced by a position observer. Passing the wrong type to the wrong stage is a compile error, not a runtime bug.
