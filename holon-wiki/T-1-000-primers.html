What This Document Is

An honest extraction of the conceptual chain from the six primers on the algebraic-intelligence.dev site. These primers explain why the holon-rs primitives exist before the code explains how. They are the conceptual root of the dependency graph.

Each primer is summarized below with: what it introduces, the key claims/definitions (quoted verbatim where possible), what holon-rs primitive it maps to, and what downstream concept it motivates.

Primer 000: VSA/HDC — A Working Introduction (162 lines)

What It Introduces

The foundational mathematical framework: Vector Symbolic Architectures / Hyperdimensional Computing. The minimum you need to understand everything else in the project.

Key Claims and Definitions

  • Hypervector: "A very large vector of numbers — thousands of dimensions of values in {-1, 0, 1}."
  • Near-orthogonality: "Random vectors are nearly orthogonal. Two random bipolar vectors have an expected cosine similarity of 0, with standard deviation ~1/√d where d is dimensionality."
  • Dimension guidance: "The literature (Kanerva) suggests at least 10,000 dimensions for the guarantees to hold comfortably." Holon's experiments ran at 16,384 and 4,096.
  • MAP variant: "Holon follows MAP. ... bipolar vectors {-1, 0, 1}, element-wise multiplication for binding, element-wise addition for bundling."
  • Self-inverse binding: "Binding is self-inverse. bind(bind(A, B), A) = B. Unbinding uses the same operation as binding."
  • Three operations, everything built from these: "VSA defines three core operations. Everything in Holon is built from these."
  • Bind — element-wise multiplication. Encodes role-filler relationships.
  • Bundle — element-wise majority vote. Superposes bound pairs into a document vector.
  • Query/Probe/Search — cosine similarity, all the same operation at different abstraction levels.
  • Structure over semantics: "VSA encodes structure: the relationship between a role and a filler, regardless of what those things mean."
  • What Holon adds (preview): Deterministic atom generation, multiple scalar encoding paths, the algebraic primitives (prototype, difference, negate, amplify, blend), accumulators with decay, engram memory.

Maps To

  • holon-rs kernel: VectorManager, bind, bundle, cosine_similarity (core operations)

Motivates

  • Everything downstream — all algebra ops, memory layer, encoding stack, the entire project.

Primer 001: Atoms, Vectors, and the Encoding Stack (231 lines)

What It Introduces

The complete encoding pipeline: how structured data (JSON, events, documents) becomes a single point in high-dimensional vector space. The stack is atomization → binding → bundling.

Key Claims and Definitions

  • Three scalar encoding paths:
  • String atomization — "The value is treated as an opaque identifier and mapped to a unique high-dimensional vector called an atom." No proximity implied between string atoms.
  • Magnitude-aware encoding ($linear, $log) — "For numeric values where proximity should be geometrically reflected." Interpolation between deterministic orthogonal basis vectors.
  • Temporal encoding ($time) — "For timestamps where the relevant similarity is cyclical." Decomposes into circular components (hour-of-day, day-of-week, month).
  • Deterministic atom generation: "SHA-256(atom_string) XOR global_seed → deterministic 4096-dimensional vector. Same atom, same language implementation, any machine, any process: identical vector."
  • Choosing wrong encoding corrupts geometry: "Encode a byte count as a string atom and you've thrown away the ordering that makes '3x normal rate' detectable."
  • List encoding modes: positional (default), bundle, chained, ngram. "The current set covers the cases that came up in the experiments; what's right for a given domain is a design choice, not a constraint."
  • Sets vs bundle-mode lists: "A set encoding and a bundle-mode list encoding of the same elements are not similar to each other. It's a type tag baked into the geometry."
  • Document vector as structural fingerprint: "The document vector is simultaneously a hash function and a queryable fingerprint." Same document → identical vector. Different documents → cosine similarity reflects structural proximity.
  • Hash function with geometric properties: "Regular hash functions map inputs to fixed-size integers where the output is opaque. Holon's atom vectors map inputs to high-dimensional vectors where the output has exploitable geometry."
  • Codebook is a cache, not source of truth: "'Codebook' appears in Holon's source and docs to mean something much narrower: the runtime cache of already-computed atom vectors. Pure performance optimization."
  • Claimed contribution — hash-function codebook: "Standard VSA systems require a pre-shared codebook. Holon eliminates this entirely by deriving atom vectors deterministically from a hash function. ... We haven't found this approach documented in the VSA literature."

Maps To

  • holon-rs: Encoder, VectorManager, ScalarEncoder (with $linear, $log, $time modes), list encoding strategies
  • holon-rs XDP sidecar: cold-boot encoder with no codebook file

Motivates

  • Primer 002 (algebra ops operate on these encoded vectors)
  • Primer 004 (coordination-free deployment depends on deterministic atom generation)
  • Challenge batches (every experiment tests what can be expressed as atomize → bind → bundle)
  • Trading lab encoding layers (thought vectors built from this stack)

Primer 002: The Holon Algebra — Operations Reference (685 lines)

What It Introduces

The full set of algebraic primitives Holon provides on top of the VSA foundation (bind, bundle, cosine). Organized into sections: Core Algebra, Pattern Extraction, Extended Algebra, Vector Operations, Streaming Operations, Advanced Operations.

Key Claims and Definitions

  • Motivation: "The experiments kept running into things the base algebra couldn't express cleanly: extracting the common pattern from a set of examples, detecting what changed between two states, asking 'find this but not that,' maintaining a running sense of normal over a stream."
  • Core operations beyond VSA:
  • unbind — "Binding is self-inverse for bipolar vectors: bind(bind(A, B), A) = B." Used for field attribution / surprise fingerprint.
  • prototype — "Finds the dimensions where a set of vectors agree — the stable signal across examples, with incidental variation filtered out." Unlike bundle, discards frequency. "Challenge 002: 100% topology classification across five families with no labeled training data."
  • negate — "Removes a component's influence from a superposition." Three methods: subtract, orthogonalize, flip. "Challenge 003 (topic exclusion: differentiation but not integration, 5/5 precision)."
  • amplify — "Boosts a component's presence in a superposition." "Challenge 003: boosted matching scores from 0.22 to 0.59 (2.6×)."
  • blend — "A vector geometrically between two inputs. Soft OR."
  • difference — "Encodes the delta between two states as a vector." "Consistent transformation rules cluster at ~0.58 cosine similarity within a type vs ~0.19 across types."
  • analogy — "Transfers a learned relationship to a new context. Computes C + (B - A)."
  • Pattern extraction: resonance (sign-agreement filter), permute (circular shift for positional encoding), cleanup (nearest-atom snap), similarity_profile, complexity, invert
  • Extended algebra: attend (soft attention, no learned weights), project (subspace projection), reject (orthogonal complement — "what the subspace cannot account for"), conditional_bind (gated binding), segment (structural breakpoints)
  • Streaming: accumulate/decay — "Maintains a running float sum over a stream — not thresholded, not normalized yet. ... the accumulator preserves the fact that a field appearing in 99% of packets should dominate the baseline 99× more than one appearing once." Plus: accumulate_weighted, merge_accumulators, capacity, purity, participation_ratio, drift_rate, autocorrelate, cross_correlate
  • Algebraic explainability: "Explainability falls out of the algebra for free. ... every document vector is built from reversible operations. ... any composite vector can be decomposed back into field-level contributions using unbind." "Neural anomaly detectors can't do this. Their representations aren't decomposable."
  • Composition as the primary use model: "The primitives were designed to be combinable. The interesting applications — and most of the ones that appeared in the challenge batches — come from composition rather than individual operations. The library is a vocabulary; what you build with it is up to you."
  • Claimed contributions:
  • negate and amplify as named first-class query operators
  • difference as standalone query primitive
  • drift_rate — temporal derivative of similarity
  • coherence as attack signal
  • purity and participation_ratio on accumulators
  • Frequency-preserving accumulation for baseline learning
  • Hash-function-as-codebook (detailed in 001)
  • Algebraic explainability without a separate explainer

Maps To

  • holon-rs algebra: all named operations in the kernel library — bind, bundle, unbind, prototype, negate, amplify, blend, difference, analogy, attend, project, reject, resonance, cleanup, coherence, drift_rate, etc.
  • holon-rs streaming: Accumulator, decay, merge_accumulators, capacity, purity

Motivates

  • Primer 003 (memory layer builds on these ops for learning and persistence)
  • DDoS lab (surprise fingerprint via unbind, anomaly detection via reject/residual)
  • Trading lab (thought vectors composed from these operations, difference for state changes, coherence for regime detection)
  • Challenge batches 001–018 (every result traces back to composition of these primitives)

Primer 003: Holon Memory — Subspaces and Engrams (401 lines)

What It Introduces

The memory layer: how Holon learns geometric patterns from streaming data, snapshots them as named portable artifacts called engrams, and matches against them. Three major components: OnlineSubspace (incremental manifold learning), StripedSubspace (crosstalk-free attribution for high-cardinality data), and Reckoner (unified discriminant learner with discrete/continuous modes).

Key Claims and Definitions

  • OnlineSubspace — incremental manifold learning: "Any stream of encoded vectors occupies a region of the high-dimensional vector space. That region isn't random; it has structure." Uses CCIPCA (Candid Covariance-free Incremental PCA, Weng et al. 2003). "O(k·dim) cost per update — no matrix inversions, no batch requirements, no stored history."
  • Residual as anomaly signal: "Low residual = the subspace explains most of the vector = in-distribution. High residual = the subspace can't account for it = anomalous."
  • Capacity ceiling: "A network packet with ~7 bindings. The subspace learns these directions comfortably with k=32 components. HTTP requests ... 80–100+ leaf bindings. CCIPCA with k=32 components can't separate them." The fix is StripedSubspace, not more dimensions.
  • StripedSubspace — crosstalk-free attribution: "Distributes leaf bindings across N independent subspaces. Instead of one vector with 100 bindings, you get N vectors with ~100/N bindings each." Stripe assignment via FNV-1a hash of fully-qualified field path. "With 100 leaf bindings across 32 stripes, each stripe averages ~3 bindings."
  • Dual-signal principle: "Every successful discrimination in the holon project has required both magnitude and direction." Aggregate residual (magnitude) + residual profile (direction). "Neither alone is sufficient. Combined, they haven't failed."
  • Engram — a named memory: "An engram is a named, serializable snapshot of a trained OnlineSubspace, plus metadata." Contains: snapshot (learned subspace), eigenvalue_signature (compact shape fingerprint), surprise_profile (per-field anomaly attribution), metadata (application-defined).
  • Surprise fingerprint: "The surprise_profile ... unbinding the reconstruction residual with field role vectors to isolate per-field anomaly contributions." Concrete example: dst_port: 0.91, ttl: 0.87, bytes: 0.79 → rule generator writes (and (= dst_port 53) (= ttl 245)).
  • EngramLibrary — pattern memory bank: Two-tier matching: eigenvalue pre-filter O(k·n) then full residual scoring O(k·dim). match_spectrum enables early warning by comparing eigenvalue shapes of traffic windows against known patterns.
  • Reckoner — unified discriminant learner: "Where OnlineSubspace learns what normal looks like (anomaly detection), Reckoner learns what separates classes or predicts values." Two readout modes: discrete (N-ary classification via discriminant directions) and continuous (scalar regression via f64 pipeline). Self-evaluates via curve: accuracy = 1/N + a × exp(b × conviction).
  • Claimed contributions:
  • Engram architecture as a whole (learned subspace, not centroid, as memory artifact)
  • CCIPCA subspace learning as detection primitive (manifold-aware, not prototype-based)
  • Two-tier eigenvalue pre-filter for engram matching
  • Field attribution via VSA unbinding of anomalous component
  • Passive upgrade via engram distribution
  • Striped FQDN leaf hashing for crosstalk-free VSA attribution
  • Residual profile as dual-signal detection primitive
  • Reckoner with discrete/continuous dual readout + self-evaluating curve

Maps To

  • holon-rs memory: OnlineSubspace, StripedSubspace, Engram, EngramLibrary, Reckoner
  • holon-rs f64 pipeline: bind_f64, bundle_f64, negate_f64, cosine_f64, ScalarEncoder::encode_f64

Motivates

  • Primer 004 (engram distribution requires coordination-free encoding)
  • DDoS lab (OnlineSubspace for baselines, StripedSubspace for HTTP, engrams for pattern distribution, surprise fingerprint for rule synthesis)
  • Trading lab (Reckoner for direction prediction in discrete mode, stop distance learning in continuous mode, conviction curves for self-evaluation)

Primer 004: Coordination-Free at Scale (126 lines)

What It Introduces

The distributed systems architecture that falls out of the hash-function codebook decision. How a single early design choice eliminates an entire class of distributed coordination problems.

Key Claims and Definitions

  • No codebook: "Holon doesn't have a codebook. It never did. That was a deliberate early design decision, and it turns out to have been the right one."
  • Operational checklist — Holon: "Same seed constant in every node's config. That's it."
  • HQ/Edge model: HQ learns and mints engrams centrally. Edge receives serialized engrams and scores in real time. "The edge node loads the engram, re-derives atom vectors on demand from the same hash function, and immediately operates in the same geometric space as HQ."
  • Determinism as correctness requirement: "If the edge derived different atom vectors from the same inputs, those stored vectors would be geometrically incoherent in the edge's space. ... Determinism isn't an optimization here; it's the foundation the entire memory layer stands on."
  • Passive upgrade: "Adding a new detection capability means distributing a new engram. ... The engram is the capability; it carries the learned pattern, the threshold, and the metadata that describes what a hit means."
  • Engrams are config, not code: "Config can be swapped, rolled back, and audited as data. Code changes require a whole different process."
  • One constraint: "Stay in one language per deployment." Different language implementations produce different vectors from the same seed.
  • The broader implication: "The seed and dimensionality define a coordinate system that exists nowhere physically but is instantiated identically everywhere. ... The space is the network."
  • Stability across time: "Engrams minted today are valid indefinitely — the geometric relationships they encode don't expire. There's no model to retrain, no vocabulary to version, no stored representations that go stale."

Maps To

  • holon-rs deployment: seed parameter in VectorManager, engram serialization/deserialization, XDP sidecar cold boot
  • Architecture: HQ/edge separation in DDoS lab

Motivates

  • DDoS lab architecture (HQ learns → engram → edge scores, no coordination)
  • Trading lab deployment model (central learning, distributed scoring)
  • Primer 005 (Wat language compiles to Rust — single-language constraint)

Primer 005: The Wat Language (193 lines)

What It Introduces

The specification language for the system: s-expressions that describe programs, with automated verification that Rust implementations match specifications. Two algebras, three structural forms, host language forms, standard library, and compilation path to Rust.

Key Claims and Definitions

  • What Wat is: "Wat is a specification language. It describes programs in s-expression form — what a module reads, what it emits, what it does not read. The wat leads. The Rust follows."
  • Two algebras:
  • Algebra 1: Vector algebra — 4 generators: atom, bind, bundle, cosine. "These are defined in core/primitives.wat. Everything in the algebra ops primer — negate, amplify, difference, prototype, attend, coherence — derives from these four in std/vectors.wat."
  • Algebra 2: Reckoner coalgebra — Learning & prediction: reckoner, register, observe, predict, decay, resolve, curve, discriminant. "Five core operations, four accessors. The accumulators are private — the coalgebra is defined by its interface, not its representation."
  • Three structural forms: Products (structs with ? suffix for optional), Coproducts (enums with exhaustive match), Protocols (type classes — check-only, no dynamic dispatch).
  • No nil: "Absence is structural — optional fields use ? suffix, conditional execution uses when. There is no null value to accidentally propagate."
  • Quote is data: "'(bind role filler) is an s-expression tree, not an execution. This enables thought vectors to be specified as data structures that the encoder evaluates — the vocabulary IS a quoted program."
  • The wards: Five automated verification skills (/sever, /reap, /scry, /gaze, /forge). "When they disagree, the wat wins."
  • Size: "8 files. 652 lines. Everything the system needs to specify itself." Standard library is 128 lines across 4 files.
  • Design principle: "The stdlib enables. The application decides. ... No vocabulary. No encoding conventions. No application patterns. The boundary is the Fact interface: the language knows how to bind and bundle. The application knows what 'RSI divergence' means."

Maps To

  • holon-rs build system: Wat specification files, /scry and /forge wards, compilation to Rust
  • Trading lab: examples/enterprise.wat (315 lines reference program), 37 wat files across 4,804 lines

Motivates

  • Trading lab enterprise architecture (wat as source of truth, Rust follows)
  • Verification methodology across the project (specification-first development)

Recurring Themes

  1. Geometry encodes structure, not semantics. The algebra doesn't know what the data means. It encodes structural relationships. Similarity falls out of cosine similarity.
  1. Determinism is load-bearing. The hash-function codebook isn't an optimization — it's the correctness foundation. Without identical atom vectors on every node, engrams are geometrically incoherent.
  1. Explainability is algebraic, not post-hoc. unbind(anomalous_component, role_vec) recovers per-field anomaly contributions because the encoding was built from reversible operations. No SHAP, no approximation.
  1. Capacity is finite and real. Near-orthogonality has limits (~O(dim/log(dim))). Striping solves the practical crosstalk problem for high-cardinality data. Accumulators saturate; capacity and purity diagnose when.
  1. Composition over individual operations. "The interesting applications come from composition rather than individual operations. The library is a vocabulary; what you build with it is up to you."
  1. Honest claims. Every "likely contribution" is explicitly caveated: "We haven't done an exhaustive literature survey." "Not a claim of priority, but a flag." "If any of this maps to existing published work, we'd genuinely want to know."

Honest Assessment

The primers are internally consistent and well-structured. The conceptual chain is clear: VSA math → encoding pipeline → algebra operations → memory/learning → deployment architecture → specification language. Each primer builds on exactly the previous ones.

The claimed contributions are stated with appropriate humility and concrete references to what they enable (specific challenge batch results, specific F1 scores). The writing is unusually transparent about provenance (the VSA foundation came from a conference video + LLM conversations, not paper reading).

No conflicts or contradictions were found between primers. The only hard constraint stated across the system is the single-language-per-deployment requirement for vector compatibility — which is clearly documented and honestly presented as a non-problem for any sane deployment.

This document serves as Tier -1 in the dependency graph: the conceptual root from which all code-level tiers descend.
