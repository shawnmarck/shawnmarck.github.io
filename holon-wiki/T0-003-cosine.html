T0-003: Cosine Similarity — the only measurement in holon-rs

Status: Verified from source
Sources: holon-rs/src/kernel/similarity.rs (581 lines), wat/core/primitives.wat (lines 1–26)

What it IS

Cosine similarity is the singular measurement primitive in the holon system. The WAT spec declares it plainly:

"Measure similarity between two thoughts. The only measurement. One number."
(cosine thought direction) → Float ; [-1.0, +1.0]

The WAT spec further situates cosine within the algebraic structure:

"A commutative monoid under bundle, with bind as the group action and cosine as the measurement homomorphism."

Cosine similarity measures the angle between two vectors, independent of their magnitudes.

How it works

Core definition

The module doc comment gives the formula:

Non-SIMD path (default)

When the simd feature is not enabled, Similarity::cosine computes:

  1. Dot product via dot_raw: iterates over paired i8 elements, casts to i64, multiplies, sums.
    2. 
   a.data().iter().zip(b.data().iter())
       .map(|(&x, &y)| (x as i64) * (y as i64))
       .sum::() as f64
  1. L2 norm via Vector::norm(): sum of squared i8 values cast to i64, then square-rooted.
    2. 
   let sum_sq: i64 = self.data.iter().map(|&v| (v as i64) * (v as i64)).sum();
   (sum_sq as f64).sqrt()
  1. Division with zero-guard: if either norm < 1e-10, returns 0.0. Otherwise returns dot / (norm_a * norm_b).

SIMD path (simd feature)

When compiled with the simd feature, cosine uses simsimd::SpatialSimilarity:


let dot_ab = i8::dot(a.data(), b.data()).unwrap_or(0.0);
let dot_aa = i8::dot(a.data(), a.data()).unwrap_or(0.0);
let dot_bb = i8::dot(b.data(), b.data()).unwrap_or(0.0);
let norm_product = (dot_aa * dot_bb).sqrt();

The module header claims this gives "up to 200x faster similarity computations on supported hardware."

f64 variant

A separate cosine_f64(a: &[f64], b: &[f64]) exists in primitives.rs for when vectors are already in float space. Same formula, different operand type.

Data types involved

Cosine similarity operates on Vector, which the code defines as:

"A high-dimensional vector with bipolar elements {-1, 0, 1}. Internally stored as i8 for memory efficiency."

Return value semantics

The source doc comment on Similarity::cosine states:

"Returns a value in [-1, 1] where:
- 1 means identical (parallel)
- 0 means orthogonal (unrelated)
- -1 means opposite (anti-parallel)"

Tests confirm: identical vectors yield 1.0, negated vectors yield -1.0.

All cosine variants

MethodSignatureDescription
cosine(a: &Vector, b: &Vector) -> f64Standard cosine similarity
weighted_cosine(a: &Vector, b: &Vector, weights: &[f64]) -> f64Per-dimension importance weighting
cosine_f64(a: &[f64], b: &[f64]) -> f64Cosine on raw f64 slices

weighted_cosine multiplies each dimension's value by its weight before computing the standard formula. The source describes it as: "Applies weights element-wise before computing cosine similarity."

Statistical significance transform

The module provides a z-score conversion:

"In high dimensions, random bipolar vectors are nearly orthogonal with cosine similarity ≈ N(0, 1/√d). This converts a raw similarity into a z-score."

Implementation: similarity * sqrt(dimensions).

Cosine is NOT the only metric in the implementation

Despite the WAT spec calling it "the only measurement," the Rust implementation provides a full Metric enum with 8 members dispatched through Similarity::compute:

  • Cosine — "most common, scale-invariant"
  • Dot — "raw inner product"
  • Euclidean — distance converted to similarity via 1 / (1 + distance)
  • Manhattan — same conversion scheme
  • Hamming — "fraction of positions that agree"
  • Overlap — "agreement among non-zero positions"
  • Agreement — (agreements - disagreements) / dimension
  • Chebyshev — L∞ distance converted via 1 - (max_diff / 2)

Additionally: weighted_euclidean, minkowski, and minkowski_distance are implemented but not in the Metric enum.

The WAT spec's "only measurement" statement refers to the conceptual primitive — cosine is the canonical measurement in the algebraic specification. The Rust implementation exposes more metrics for practical flexibility.
