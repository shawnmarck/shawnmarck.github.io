T0-005 · VectorManager HOW

T0-005 · VectorManager

What it IS

VectorManager is the deterministic atom-to-vector registry inside the Holon kernel. It guarantees that the same atomic value always produces the same vector, given the same seed and dimensionality.

Source: holon-rs/src/kernel/vector_manager.rs (325 lines)

"This module provides the core guarantee of Holon: The same atomic value ALWAYS produces the SAME vector."

It is a #[derive(Clone)] struct wrapping two Arc>> caches — one for atom vectors, one for position vectors — backed by a SHA-256 → ChaCha8Rng deterministic pipeline.

Structure


pub struct VectorManager {
    dimensions: usize,
    global_seed: u64,
    cache: Arc>>,
    position_cache: Arc>>,
}

Two constructors:

  • new(dimensions) — seed defaults to 0
  • with_seed(dimensions, global_seed) — explicit seed for reproducibility

The Arc> caches are shared across clones: all clones see the same underlying store.

How it works

Atom vector computation (get_vector)

  1. Check the cache read-lock for a hit. Return immediately if found.
  2. Otherwise, call compute_vector(atom):
  • Take the first 8 bytes of the hash as a u64 seed
  • Seed a ChaCha8Rng from that seed
  • Generate dimensions values, each rng.next_u32() % 3, mapped to {-1, 0, 1}
  • Wrap in a Vector
  1. Write-lock the cache and insert the result.
"Generate random bipolar vector with {-1, 0, 1}. Match Python's distribution: 1/3 probability each."

The distribution is uniform: each dimension independently takes -1, 0, or 1 with equal probability.

Position vector computation (get_position_vector)

Position vectors are atom vectors with a synthetic key. The method formats the position as "__pos__{position}" and delegates to compute_vector. This matches the Python implementation's _position_to_seed().

"Position vectors are seeded with __pos__{position} to match the Python implementation's _position_to_seed()."

Position vectors live in a separate position_cache keyed by i64, so get_position_vector(0) is never confused with get_vector("0").

Codebook export/import

  • export_codebook() → HashMap> — serializes the atom cache
  • import_codebook(codebook) — restores cached vectors; existing entries preserved, imported entries overwrite on conflict

This enables persisting the vector mapping across process restarts or sharing identical mappings between distributed nodes — even with a different seed, imported vectors are served from cache and never recomputed.

Key properties

PropertyDetail
DeterminismSame (dimensions, seed, atom) → identical vector, always. Verified by test test_verify_determinism across 5 atoms × 3 repetitions.
Seed isolationDifferent seeds → different vectors for the same atom.
Atom isolationDifferent atoms → different vectors (with overwhelming probability).
Position isolationget_position_vector(n) ≠ get_vector(str(n)).
Cache sharingClones share Arc-backed caches; thread-safe via RwLock.
Bipolar distributionUniform {-1, 0, 1} — 1/3 probability each dimension.
Hash pipelineSHA-256(global_seed ∥ atom)[0..8] → ChaCha8Rng → bipolar vector.
