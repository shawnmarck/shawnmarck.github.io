Disposable Machine EXPLAIN

TL;DR

The DNA/Phenotype Split

The architecture enforces a strict separation between what the system is and what it currently looks like:

LayerRoleMutability
T-1-004-lab-guide (GUIDE.md)Specification — declares every struct, enum, dependency, and data flowChanges only when the thesis changes
wat filesIntermediate representation — Scheme-like s-expressions that implement what the guide declaresRegenerated from guide changes
Rust implementationPhenotype — compiles and runs, but carries no permanent authorityDisposable. Grown, run, discarded.

The chain of derivation is one-directional: Guide → wat → Rust. The Rust never feeds back into the guide. When layers diverge, the guide is right — by declaration.

This is stated explicitly: "Each section declares its dependencies. The order of sections IS the build order." The guide is not documentation about the code. The guide is the code's reason to exist.

Why Disposable?

Legacy is Rot

Most software systems accumulate cruft: dead code paths, workaround patches, architectural compromises that made sense once but don't anymore. Each one is a hidden coupling — change one thing, break another you forgot existed. The trading organism avoids this by refusing to accumulate.

The Machine is a Function


f(state, candle) → state

One tick of the clock. All 11 sub-circuits compose into this single pure function. A pure function has no hidden state, no legacy paths, no accumulated decisions that can't be traced from the specification. If you can write down f, you can rebuild f.

The Organism Metaphor

The guide describes the system as a "machine that measures thoughts against reality." An organism doesn't patch itself incrementally — it grows from genetic instructions. When a cell dies, a new one differentiates from the same DNA. The trading organism works the same way:

  • DNA = the GUIDE.md (specification)
  • Differentiation = compilation from wat to Rust
  • Phenotype = the running binary
  • Death and regrowth = discard the Rust, regrow from the guide

How It Works in Practice

Construction Order Is Specification Order

The guide's struct hierarchy (lines 282–299 of the GUIDE) isn't a reference — it's the build sequence. Each struct declares what it depends on. You construct them in that order. There is no other source of truth for "what depends on what."

The Five Primitives

Everything is built from five holon-rs primitives: atom, bind, bundle, cosine, reckoner. One learning mechanism (Reckoner). One accountability surface (Curve). The entire organism — market perception, prediction, distance estimation, trade management, learning — differentiates from these five primitives as described in the guide.

No Bootstrap, No Migration

The system starts ignorant. Every Reckoner begins with zero experience. Papers fill the reckoners, experience grows, the Treasury starts listening. There is no migration path from an old version. There is no serialized state to load. You build from the guide, you run, you learn from scratch. The learning is the state — and learning is always acquired fresh.

Discard and Regrow

When you need to change the system:

  1. Update the wat files — regenerate the s-expression implementations from the changed guide
  2. Rebuild Rust — compile the new phenotype
  3. Run — the organism starts ignorant and learns again

No diff against old code. No migration scripts. No "what did we change last time." The guide is the single source of truth, and the implementation is always a fresh expression of it.

The Circuit as Phenotype Map

Together they form a complete specification:

  • GUIDE = semantic specification (what each component means, what it depends on)
  • CIRCUIT = structural specification (how signals flow, what the topology looks like)
  • Rust = executable phenotype (the running machine — disposable)
