Wards — the principle of scoped, disposable cognition EXPLAIN

TL;DR: Wards are automated verification skills — named, invoked, and discarded. They are how the holon system enforces architectural honesty without accumulating permanent process. Each ward is a cognitive scope: it receives a specific task, executes against the codebase, reports findings, and is dissolved. The system that polices the specification is itself disposable — the guide is the DNA, the wards are the immune response.

What it is

"The wards: /sever, /reap, /scry, /gaze, /forge — five automated skills. Runes for conscious suppression." — T-1-001-story, Series 006-004
"Wat is a specification language. It describes programs in s-expression form — what a module reads, what it emits, what it does not read. The wat leads. The Rust follows." — T-1-000-primers

Scoped Cognition

Wards embody a key architectural principle: cognition should be scoped and disposable, not accumulated.

Each ward invocation creates a fresh cognitive context — an agent that knows exactly what it needs for its task and nothing more. The /ignorant ward is the purest expression of this: it reads the guide with zero prior context, simulating a stranger encountering the system for the first time. If the stranger can reconstruct the system from the document alone, the document is honest.

"32 passes of the ignorant reader on the guide. Findings dropped from 18 to 0." — T-1-002-book, Act IV

This produced the fixed point:

"The guide is the fixed point. f(guide) = guide. The ignorant ward applied to the guide produces the guide. No changes needed." — T-1-002-book

Ward-Enforced Constraints

Wards enforce constraints at the language level, not just through post-hoc review. In the wat-language, parallelism constraints are ward-enforced at compile time:

"pmap: the mapped function must be pure (no set!, push!, inc!). Ward-enforced at compile time." — T1-001-wat-language

This is the pattern: wards are not suggestions or linters. They are structural guarantees. The wat language defines what purity means; the wards verify that code conforms. When code violates the spec, it does not compile.

Quantitative Impact

The ward system produced measurable results during the Forging phase (Series 006-005):

  • 213 phantom runes (forms referenced in code that don't exist in the language) dissolved to zero
  • The LLM, when asked to fix phantoms, "replaced phantoms with new phantoms (6 new ones) — fluently, and wrong" — demonstrating why automated verification, not trust, is required
  • 405 lines of dead code removed from a single file
  • 35 visual ghosts eliminated
  • 19 dead thoughts reaped
  • Throughput improved from 139 to 222 candles/sec — "from removing lies, not from optimization" — T-1-001-story, Series 006-004
"Honest architecture produces honest results." — T-1-001-story, Series 006-004

Wards as Vocabulary Guardians

  • /sever removes code that names things that aren't real
  • /scry ensures the specification and implementation use the same vocabulary
  • /assay measures whether a named concept has the substance its description claims
  • /temper finds concepts that are almost useful — efficient waste that looks like signal but isn't

The 213 phantom runes dissolved by the wards were vocabulary errors: the LLM had written code referencing forms that do not exist in the language. The wards caught every one. The LLM then generated new phantoms when asked to fix them — proving that verification must be mechanical, not interpersonal.

Relationship to the Disposable Machine

This is not metaphor. The architecture specifies:

"The disposable machine: wat can be thrown away and regenerated from the guide." — T-1-001-story, Series 006-006

If the wat (and by extension the Rust) can be regenerated, then the wards that verify them must also be regenerable. They are not hand-curated tooling — they are skills invoked from the guide's specification of what correctness means. The guide describes the language; the wards verify conformance; both are disposable outputs of the same specification process.
