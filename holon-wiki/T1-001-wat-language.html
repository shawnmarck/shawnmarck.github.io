The wat Language — forms, evaluation, and structural types DESCRIBE

The wat Language — forms, evaluation, and structural types

Verified leaf: T1-001

Sources: wat/LANGUAGE.md (341 lines), wat/core/structural.wat (115 lines)

What wat is

Wat is "an s-expression language for algebraic cognition." It is not a general-purpose language, not Turing complete, and not interpreted at runtime — it "specifies Rust implementations." It is "Lisp with a purpose: algebraic cognition from named thoughts."

  • "A specification language for Holon programs."
  • "A shared language between human intuition and machine implementation."
  • "The intermediate representation between thought and execution."

Grammar

EBNF grammar from LANGUAGE.md lines 7–16:


program    = form*
form       = atom | number | string | list | typed-param | comment
atom       = letter (letter | digit | '-' | '?' | '!' | '>')*
number     = digit+ ('.' digit+)? 
string     = '"' char* '"'
list       = '(' form* ')'
typed-param = '[' atom ':' atom ']'
comment    = ';' char* newline

A form is the fundamental unit: an atom, number, string, list, typed parameter, or comment.

Type annotations (optional)

"Parseable but not enforced." The source says: "The annotations are metadata on a free algebra — they name which carrier set a variable lives in."

Two forms:

  • Typed parameter: [name : Type]
  • Return type: : Type after the parameter list

"Tooling may extract annotations when present. The language never demands them." "Types are documentation, not enforcement. The algebra doesn't type-check — the curve validates."

Host language forms

Wat "inherits standard forms from the host" Lisp. These are enumerated in the source and include:

Arithmetic: +, -, *, /, abs, sqrt, mod, max, min, round, clamp, exp, ln, signum, f64-infinity, f64-neg-infinity

Comparison: =, !=, >, <, >=, <=

Logical: and, or, not, true, false

Sequencing: begin

Iteration: for-each, map, filter, filter-map, fold, fold-left, count, pmap, pfor-each

The source specifies that pmap and pfor-each are parallel variants: "Semantically identical — same results, same order. The parallelism is a permission, not a directive." Constraints:

  • pmap: "the mapped function must be pure (no set!, push!, inc!). Ward-enforced at compile time."
  • pfor-each: "each element's mutations must be disjoint."
  • "No pfold: a fold is inherently sequential." The pattern is (fold f init (pmap g xs)).

Collections: list, cons, len, length, nth, first, second, rest, last, last-n, take, append, take-last, empty?, reverse, sort, sort-by, flatten, range, unzip, zeros, member?, some?, quantile, apply

Note on zeros: "nullary: zero vector at dims; (zeros n): zero-filled array of size n"

Maps: map-of, get, assoc, keys, dissoc. map-of takes flat key-value pairs. assoc is variadic with sequential semantics — "later entries see earlier changes."

Queues: deque, push-back, pop-front

Strings: format, substring

Control: let (with destructuring), let*, define, if, when, when-let, cond (with else catch-all), match, lambda

Quote: '(...) or (quote ...). The source emphasizes: "The s-expression IS the tree." "Recursive structure is natural." "No special type needed. The parentheses ARE the tree."

Optionals: (Some value), None — maps to Rust's Option.

Mutation: set! (single and indexed forms), push!, pop!, inc! — these "map to &mut self" in Rust.

Pipes: make-pipe, send, recv, try-recv, select

Fan-out: make-topic

Threads: spawn, join

Concurrency model: pipes, topics, threads

Pipes

"Pipes are CSP (communicating sequential processes)."

make-pipe takes :capacity (a number or :unbounded) and :carries (a type declaration). Returns a (tx, rx) pair. "Destructure at creation."

  • send: blocking write.
  • recv: blocking read → value.
  • try-recv: non-blocking → (Some value) or None.
  • select: multiplexes N rx ends → (Some (idx value)) or :closed. "Parks when idle — zero CPU."

Topics

make-topic wires 1→N fan-out over existing pipes. "A topic is a write-only value." The source states: "The caller's send IS the fan-out — synchronous, no thread, no queue." Maps to a Rust struct holding Vec> with a for loop.

Threads

  • spawn: "run a function on a thread" → Handle. "A process is a function."
  • join: "block until all handles in a list complete" → list of return values. "Always takes a list. Always returns a list."

Design principle

"No defpipe. No defprocess. No defservice. Pipes are values. Topics are values. Processes are functions." Everything is regular functions using the seven forms listed above.

Core forms (corelib)

The source describes "Two algebras and one structural form. The algebras transform values. The structural form carries them."

The free algebra (vector operations)

  • (atom "name") → Vector — "Naming"
  • (bind role filler) → Vector — "Composition" — "two things become one relationship"
  • (bundle fact1 fact2 ...) → Vector — "Composition" — "many facts become one thought"
  • (cosine thought discriminant) → Float — "Measurement" — returns value in [-1.0, +1.0]

Structural forms

From structural.wat: "The ambient category's three constructions: Product (struct), Coproduct (enum), Type class (defprotocol)."

The source states these are "orthogonal to the vector algebra and journal coalgebra." "A struct cannot be bound, bundled, or cosined." "Protocols never enter the algebra."

Struct (product type)

(struct name field1 field2 ...)

  • "Untyped: the wat struct names the shape, the Rust struct names the types."
  • "Opaque: no destructuring, no pattern matching on fields. Projections only."
  • "A field name ending in ? is optional" — maps to Option in Rust, initialized to None.
  • Constructors follow the make- convention: (make-broker ...).

Field access: (:field record) → value — "keyword as function."

Functional update: (update record :field1 value1 ...) → record. "Variadic, parallel semantics — all field expressions evaluate against the ORIGINAL record."

Enum (coproduct / sum type)

(enum name variant1 variant2 ...)

  • "Exactly one of several alternatives."
  • "Simple variants are keywords. Tagged variants carry data."
  • match must be exhaustive — "missing an arm on a closed set is a spec violation."
  • "Wildcard _ is forbidden on closed enums — it defeats exhaustiveness."

Newtype

(newtype name inner-type) — "A newtype is about MEANING, not structure." "Maps to Rust's tuple struct: struct TradeId(usize)."

Protocol (type class)

(defprotocol name (fn-name [params] "docstring")) — "Check-only — no dispatch machinery."

(satisfies struct-name protocol-name :fn1 impl1 :fn2 impl2 ...) — "Maps ALL protocol functions — exhaustive, explicit."

The source specifies:

  • "One satisfies per (struct, protocol) pair. Duplicate = error."
  • "No convention-based inference. The mapping IS the spec."
  • "The forge checks: does impl-fn exist? Correct arity?"
  • Maps to Rust trait and impl Trait for Struct.
