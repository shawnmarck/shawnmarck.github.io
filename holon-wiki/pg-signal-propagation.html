Signal Propagation — How Learning Flows Backward Through the Organism EXPLAIN

The Resolution — What Triggers Learning

Every settled paper produces a Resolution struct carrying 15 fields. The three resolution paths:

PathTriggerOutcomeSignal destinationPosition batch?
Stop fires before trail crossesPrice hits safety stopViolencemarket_signals → market observerNo
Trail crosses (first time)Trailing stop reachedGrace signalmarket_signals → market observerRunner begins
Runner finishesReversal hits trailing stopGrace resolutionrunner_resolutions → position observerYes — deferred batch

Key ordering: Violence is checked before Grace. If both fire on the same tick, Violence wins. (T0-018, §7.7)

What Each Observer Learns

Market observer receives the market_thought — the raw market anomaly before composition with position data. Per Proposal 024:

"market observer learns from the anomaly, not composed thought" — T0-018, line 112

Position observer receives the position_thought — its own encoded facts only. Per Proposal 026:

"position learns from position_thought, not composed" — T0-018, line 39

This separation ensures each observer learns from what it actually produced, not from downstream mixing.

PropagationFacts — The Return Value


pub struct PropagationFacts {
    pub market_idx: usize,       // Which market observer should learn
    pub position_idx: usize,     // Which position observer should learn
    pub direction: Direction,    // For the market observer (Up/Down label)
    pub composed_thought: Vector,// For both observers
    pub market_thought: Vector,  // Proposal 024: market learns from anomaly
    pub position_thought: Vector,// Proposal 026: position learns from this
    pub optimal: Distances,      // For the position observer
    pub weight: f64,             // For both observers
}

— T0-018, §5

The Post collects these from all brokers and routes them to the correct observers via mailbox channels. The broker does not talk to observers directly.

"Values up — the broker does not talk to observers directly." — T0-018, §5

The Three Propagation Steps

Step 1 — Track Record


match outcome {
    Outcome::Grace => { self.cumulative_grace += weight; self.grace_count += 1; }
    Outcome::Violence => { self.cumulative_violence += weight; self.violence_count += 1; }
}
self.trade_count += 1;

— T0-018, §7.8 (Step 1)

Pure accounting. Cumulative grace/violence values and counts feed the gate decision.

Step 2 — Scalar Accumulators Learn


self.trail_accum.observe(optimal.trail, outcome, weight, scalar_encoder);
self.stop_accum.observe(optimal.stop, outcome, weight, scalar_encoder);

— T0-018, §7.8 (Step 2)

The accumulators observe hindsight-optimal distances (what trail/stop would have been ideal), not the distances that were actually used. This teaches the distance cascade what to aim for.

Step 3 — Dollar P&L and Expected Value

Against a $10,000 reference capital including swap fees on entry and exit:

Grace: net = weight × 10,000 − entry_fee − exit_fee (weight = excursion fraction)

Violence: net = −(weight × 10,000 + entry_fee + exit_fee) (weight = stop distance)

EMA update (α=0.038, half-life ~50 papers):


avg_grace_net   = (1 − 0.038) × avg_grace_net   + 0.038 × net
avg_violence_net = (1 − 0.038) × avg_violence_net + 0.038 × net
expected_value   = grace_rate × avg_grace_net + (1 − grace_rate) × avg_violence_net

— T0-018, §7.8 (Step 3); pg-broker (§The Gate)

This expected value drives the gate: brokers with negative EV are suppressed from proposing funded trades.

Market Observer Learning Path

When PropagationFacts arrive at a market observer's mailbox via ObsLearn:

  1. Map direction to label — Direction::Up → index 0, Direction::Down → index 1
  2. Check correctness — compare last_prediction against actual market direction
  3. Reckoner learns — reckoner.observe(market_thought, label, weight) feeds the labeled anomaly to the discrete Reckoner, updating its accumulated prototypes
  4. Conviction curve updates — a second predict+resolve cycle updates the accuracy = 1/N + a·exp(b·conviction) fit, self-calibrating the observer's confidence
  5. Engram gate evaluates — accumulates win rate and discriminant state quality; observers that prove themselves get knowledge gated into long-term memory
"Its own teacher. The market is the judge." — T0-013

— pg-market-observer (§Learning)

Position Observer Learning Path

When PropagationFacts arrive at a position observer's mailbox via PositionLearn:

  1. Pre-learning prediction — query each reckoner for what it currently predicts
  2. Trail reckoner observes — trail_reckoner.observe_scalar(position_thought, optimal.trail, 1.0)
  3. Stop reckoner observes — stop_reckoner.observe_scalar(position_thought, optimal.stop, 1.0)

Weight is always 1.0 — each observation counts once. The "optimal" distances come from hindsight simulation that sweeps candidate values against actual price histories.

— pg-exit-observer (§Learning)

Deferred Batch Training — Runner Learning

When a paper signals Grace (trail crosses), it becomes a runner. The broker accumulates a RunnerHistory — parallel arrays of composed thoughts, distances, and prices at each candle while the runner is active.

When the runner resolves, compute_position_batch() produces deferred training data:

  1. Compute optimal stop via full simulation over the runner's price history
  2. Suffix extremum pass — O(n) backward scan for the best remaining price from each candle
  3. Per-candle optimal trail from the suffix extremum
  4. Sparse sampling — emit a training sample only when optimal trail changed by >10% relative, avoiding redundant samples
  5. Weight = the optimal trail value (residue that distance would capture)

Each batch entry: (thought_at_candle, optimal_distances_at_candle, actual_distances_at_candle, weight). Included in the Resolution as position_batch and routed to the position observer by the Post.

"Only emit a training sample when the optimal trail changed by >10% relative. If the answer is the same candle over candle, there's nothing to learn." — T0-018, §8.2

— T0-018 (§3, §8.2); pg-broker (§Runners and Deferred Training)

Direction Flip — Forced Grace Propagation

When the market observer's active direction flips (was Up, now Down), close_all_runners() force-resolves all active runners with Outcome::Grace. Non-runner papers that haven't resolved survive the flip and hit their stop naturally. This ensures learning signals propagate promptly when regime changes, rather than waiting for runners to expire organically.

— T0-018, §7.6; pg-broker (§Direction Flip)

Verified Properties

  1. No shared mutation. propagate() returns PropagationFacts. tick_papers() returns Vec. The broker never touches observer state. (T0-018, §12)
  2. Thought separation is enforced. Market observer learns from market_thought; position observer learns from position_thought. Neither sees the composed thought for learning purposes. (Proposals 024, 026)
  3. Runner history is scoped. Created at Grace signal, accumulated per-candle, consumed at runner resolution. Keyed by paper_id, preventing stale data. (T0-018, §12)
  4. Sparse sampling avoids noise. Only meaningful changes (>10%) in optimal distances generate training samples. (T0-018, line 556)
  5. All learning signals are typed values on wires. No Mutex, no shared state, no direct observer mutation. (pg-services, pg-programs)
