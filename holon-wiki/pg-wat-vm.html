wat-vm — the training harness EXPLAIN

TL;DR#

wat-vm is a standalone Rust binary (846 lines at src/bin/wat-vm.rs) that replays historical candle data through a parallel grid of market observers, position observers, and brokers so they can learn. It is emphatically not a live trading system — its own CLI summary says it best: "Candles in, observers learn, results discarded." It is the training/learning harness, the "second heartbeat" of the Holon system.

Composite wiki page. Sources: T0-017 (wat-vm + chain types), T0-016 (services/messaging), T1-001 (wat language).

Architecture overview#

flowchart TD
    main_thread["Main thread"] -->|"candles"| market_obs["N market observers"]
    market_obs -->|"fan-out"| position_obs["M position observers"]
    position_obs -->|"chain"| brokers["N×M brokers"]
    brokers -.->|"learn signals"| market_obs
    brokers -.->|"learn signals + Trade updates"| position_obs
  • N market observers encode candles into vectors, detect anomalies, predict direction.
  • M position observers combine market signals with position features, compute stop distances.
  • N×M brokers evaluate position entries, manage trade lifecycle, feed learn signals back.
  • Default configuration: N=6 market lenses, M=2 position lenses → 23+ threads for a single stream.

Thread budget#

N + M + (N×M) brokers + database driver + cache driver + console driver. With defaults (6+2+12+3) = 23 threads minimum. Multiple streams or internal service threads push it higher — the project docs reference "30+ threads."

CLI interface#

wat-vm --stream SOURCE:TARGET:PATH [--max-candles N] [--dims D] [--recalib-interval R] [--swap-fee F]
Flag Default Purpose
--stream (required, repeatable) SOURCE:TARGET:PATH — e.g. binance:BTCUSDT:data/btc.csv. Multiple streams processed sequentially.

--recalib-interval 500 Observations between recalibrations (see Reckoner).
--swap-fee 0.0010 (10 bps) Venue fee fraction.

Startup sequence#

  1. Parse args. Derive observer counts from config::MARKET_LENSES and config::POSITION_LENSES. Compute num_brokers = N×M, noise_floor = 5.0 / sqrt(dims).
  2. Console pool. Create N+M+(N×M)+streams+1 handles. Claim main + stream handles.
  3. Database. SQLite at runs/wat-vm_<timestamp>.db. Unbounded queues merged via mailbox. Batch-insert driver (batch size 100, telemetry every 5s).
  4. Encoding cache. VectorManager + ScalarEncoder → LRU cache (256K capacity) for ThoughtAST → Vector. Rate-gated telemetry via database sender.
  5. Reserve handle pools. Pre-allocate console/cache/db handles for position observers and brokers; market observers get whatever remains. Pools assert finish() — orphan detection prevents deadlocks.
  6. Learn queues. N×M ObsLearn queues (fan-in to each market observer's mailbox), M×N PositionLearn queues, N×M TradeUpdate queues.
  7. Wire market observers. For each of N: bounded candle input (capacity 1), M bounded fan-out Topic queues, spawn market_observer_program.
  8. Wire position observers. Transpose [mi][ei] receiver grid into [ei][mi]. For each of M: build N PositionSlots, spawn position_observer_program.
  9. Wire brokers. Flatten 2D learn-tx grids. For each of N×M: assign output_rx, learn txs, trade tx. Spawn broker_program.
  10. Build pipelines. For each --stream: open CandleStream, create IndicatorBank.
  11. Run main loop.
  12. Shutdown.

Source: T0-017, "Startup sequence (main)" section.

Main loop#

Per-stream, sequential candle replay:

while count < max && !STOP {
    let candle = pipeline.stream.next();      // raw OHLCV from file
    let candle = pipeline.bank.tick(&ohlcv);  // enrich with indicators
    for each market observer candle_tx:
        send ObsInput { candle, window, encode_count };
    every 100 candles: log throughput;
}

Key design choices:

  • Fire-and-forget. The main thread pushes candles to all N market observers and never waits for results. It is the sole candle producer in a push architecture.
  • ObsInput carries the enriched candle, a window (Arc<Vec<Candle>>, currently length 1), and a global monotonic encode_count.
  • IndicatorBank::tick() enriches raw OHLCV into a full Candle struct.

Source: T0-017, "The main loop" section.

Data flow: the chain types#

The grid communicates through two wire types defined in src/programs/chain.rs (40 lines). The module's design principle: "Each stage produces a new value. The type IS the proof of which stage produced it. No methods. No behavior. Values on the wire."

MarketChain (Clone)#

Produced by market observers, consumed by position observers via Topic fan-out.

Field Type Meaning
candle Candle Enriched candle
window Arc<Vec<Candle>> Recent candles (currently length 1)
encode_count usize Global monotonic counter
market_raw Vector Raw encoding of market features
market_anomaly Vector Deviation from noise subspace
market_ast ThoughtAST The AST that was encoded
prediction Prediction Direction (Up/Down)
edge f64 Confidence score

MarketPositionChain (not Clone)#

Produced by position observers, consumed by brokers via unbounded queues. Flattens all MarketChain fields (duplicated, not borrowed — self-contained, no shared references) and adds position-specific fields:

Field Type Meaning
position_raw Vector Position observer's raw encoding
position_anomaly Vector Position observer's anomaly
position_ast ThoughtAST Position observer's AST
position_distances Distances Trailing stop and safety stop distances
(all MarketChain fields) (duplicated) Renamed: prediction→market_prediction, edge→market_edge

The deliberate lack of Clone and reference-free layout makes this struct large but ownership-simple — a "values on the wire" design.

Source: T0-017, "Chain types" section.

Signal handling#

A single static AtomicBool named STOP. A C signal handler (via libc::signal) sets it on SIGINT/SIGTERM. The main loop reads it each iteration. Nothing else touches it. Simple, correct, no async machinery.

Shutdown sequence#

Ordered teardown — critical for correctness:

  1. Drop candle senders → market observers see disconnect on their input queues.
  2. Join market observer threads → collect trained observers, print experience/resolved counts.
  3. Drop topic handles → topics drain remaining messages to position observers.
  4. Join position observer threads → collect trained observers, print trail/stop experience.
  5. Broker queues disconnect automatically when position observers drop their senders. Join broker threads → print trade counts, grace rate, expected value, open papers.
  6. Print database path.
  7. Print cache summary (hits/misses/rate).
  8. Drop stream handles. Join cache driver (must join before db — its emit closure holds a db_tx). Join db driver. Drop main handle. Join console driver.

Source: T0-017, "Shutdown sequence" section.

Messaging primitives#

Three primitives, zero Mutex. All built on crossbeam::channel. Concurrency through message passing alone.

Primitive Pattern Mechanism Mutex?
Queue (bounded/unbounded) 1→1 point-to-point crossbeam::channel No
Topic 1→N fan-out Bounded input → spawned thread → clone() to N output queues No
Mailbox N→1 fan-in Spawned thread → crossbeam::Select over N inputs → unbounded output No
  • Backpressure: Topics use bounded sends — one slow subscriber stalls all. Intentional.
  • Partial disconnect: Mailbox continues receiving from living producers until all disconnect.
  • Shutdown by disconnection: Dropping a sender/receiver propagates as a Disconnected error. No explicit coordination flags.

Source: T0-016, "Services (Rust) — Verification."

Relationship to the wat language#

wat-vm is implemented in Rust but its architecture is specified in wat:

  • Concurrency forms in wat (make-pipe, make-topic, spawn, join, select) map directly to the Rust Queue, Topic, and threading patterns used by wat-vm. The wat spec states: "No defpipe. No defprocess. Pipes are values. Topics are values. Processes are functions."
  • The reckoner coalgebra (wat's learning construct: observe, predict, resolve, edge-at, proven?) is the learning engine that the observers and brokers run inside their program threads.
  • Structural forms (struct, update, defprotocol) define the chain types and observer/broker records.
  • wat is "not interpreted at runtime" — it specifies Rust implementations. wat-vm is one of those implementations.

Source: T1-001, "The wat Language — forms, evaluation, and structural types."

Design principles distilled#

  1. Push architecture. Main thread is the sole candle producer. Fire-and-forget to all observers.
  2. Ownership over locking. Zero Mutex. All concurrency through channel message passing.
  3. Type-is-proof chain stages. MarketChain and MarketPositionChain are pure data structs. The type name documents which stage produced it.
  4. Graceful ordered shutdown. Teardown follows the dependency graph. Pools assert all handles claimed (orphan detection).
  5. Training only. Results are logged but discarded. No order routing, no live execution. "Candles in, observers learn, results discarded."
