Encoder Service — the distributed caching layer for thought encoding WHAT

Source: wat/encoder-service.wat (142 lines)

Guide reference: No dedicated GUIDE section found; implied by GUIDE.md lines 1439–1446 (LruCache on ThoughtEncoder)

Tier: 2 — Encoding infrastructure

What It IS

The EncoderService is a single-threaded pipe loop that owns the LRU composition cache, serving multiple concurrent callers without any mutex. Each caller gets its own EncoderHandle (a set of three pipes). The encoder thread iterates all handles once per loop, draining installs and servicing lookups.

"ThoughtEncoder cache as a single-threaded pipe loop. The encoder holds an LRU cache. Each caller gets its own pipe set (EncoderHandle). The loop iterates all pipes once per iteration. No mutex. One thread. N callers. One cache."
— encoder-service.wat, lines 1–7

It depends on thought-encoder (for the ThoughtAST type) and primitives (for Vector).

EncoderHandle — A Caller's Pipe Set


(struct encoder-handle
  [lookup  : Sender]          ; bounded 1
  [answer  : Receiver>]    ; bounded 1
  [install : Sender<(ThoughtAST, Vector)>]) ; unbounded, fire and forget

Lines 22–25. Three pipes per caller:

PipeCapacityDirectionPurpose
lookup (Sender)bounded 1caller → serviceSend a ThoughtAST for cache lookup
answer (Receiver)bounded 1service → callerReceive Some(Vector) (hit) or None (miss)
install (Sender)unboundedcaller → serviceFire-and-forget insertion of computed (AST, Vector) pair

encoder-get (lines 28–31)

Blocking lookup. Sends AST on lookup, blocks on answer, receives Option.

encoder-set (lines 34–36)

Fire and forget. Sends (AST, Vector) pair on install. Non-blocking, unbounded.

EncoderService Struct


(struct encoder-service
  [handle     : JoinHandle<()>]
  [hits       : Arc]         ; cache hit counter
  [misses     : Arc]         ; cache miss counter
  [cache-size : Arc])        ; current cache occupancy

Lines 43–47. The service does not hold sender copies — the handles ARE the senders.

"Does NOT hold sender copies — the handles ARE the senders. When all handles drop, the channels close, the cascade flows, the encoder thread exits."
— lines 39–41

Three Arc counters provide lock-free read access to hit/miss/cache-size stats from any thread.

Lines 51–122. Creates the service and N handles.

Pipe Setup (lines 56–68)

For each of n-callers, creates three pipe pairs:

  • lookup: bounded 1, carries ThoughtAST
  • answer: bounded 1, carries Option
  • install: unbounded, carries (ThoughtAST, Vector)

Sends the Sender halves into the handle; the Receiver halves are collected for the service thread.

Service Thread Loop (lines 78–119)

The spawned thread runs an infinite loop with three phases per iteration:

Pass 1 — Drain installs (lines 85–90):

Iterates all install receivers. Drains each one completely (inner loop with try-recv). Each (ast, vec) pair is inserted into the LRU cache. Updates cache-size atomic.

"Pass 1: drain ALL install pipes. Put into cache."
— line 84

Pass 2 — Service lookups (lines 93–106):

Iterates all lookup receivers. For each: tries to receive an AST. On success: looks up cache, increments hits or misses atomic accordingly, sends result back on the corresponding answer pipe. On disconnect: marks that handle as closed.

"Pass 2: service ALL pending lookup pipes."
— line 93

Shutdown check (lines 109–110):

If all handles are disconnected, the loop breaks and the thread exits.

"Shutdown: all get pipes closed"
— line 108

Idle blocking (lines 116–118):

Uses select over all open get and set pipes. Blocks until any channel has data. Zero CPU when idle.

"Block until ANY channel has data. Zero CPU when idle."
— lines 112–113

Protocol Summary

The comment block (lines 9–12) specifies the full protocol:


Caller: write AST to lookup pipe -> block on answer pipe -> receive Some/None
Caller: if None, compute locally, write (AST, Vector) to install pipe (fire and forget)
Encoder: one pass per iteration — drain installs, service lookups, select, repeat.

This implements the "values up" principle: callers compute locally on cache misses rather than requesting computation from the service. The service is purely a cache — it never computes vectors, only stores and retrieves them.

Shutdown: encoder-service-shutdown


(define (encoder-service-shutdown [svc : EncoderService]) : ()
  (join (:handle svc)))

Lines 129–131. Requires that all handles have already been dropped (callers gone). Joins the thread handle.

"The cascade must have already closed all handles."
— line 125

Accessors

Three lock-free read functions (lines 135–142):

  • hit-count → loads Arc for hits
  • miss-count → loads Arc for misses
  • cache-len → loads Arc for current cache occupancy

Relationship to ThoughtEncoder

The ThoughtEncoder (thought-encoder.wat) declares its compositions field as "optimistic cache (LRU moved to EncoderService)". This service IS that LRU cache. The ThoughtEncoder holds a local per-candle working set; the EncoderService holds the persistent cross-candle LRU cache shared across all N observer threads.

The GUIDE (line 1440) declares the compositions field as LruCache and explains the eventual-consistency model (lines 1441–1446): "miss on candle N, hit on N+1". The EncoderService implements this: callers install computed values via fire-and-forget pipes, and the service drains installs on its next loop iteration before servicing any new lookups.
