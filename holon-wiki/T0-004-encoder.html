T0-004 — Scalar Encoding HOW

T0-004 — Scalar Encoding

What It Is

The scalar encoding subsystem maps continuous numeric values (prices, rates, timestamps, temperatures) into fixed-width bipolar vectors ({−1, 0, +1}^dim) so that similar values produce similar vectors. It lives in two places:

  1. kernel/scalar.rs — standalone ScalarEncoder struct with its own deterministic random base/ortho vectors.
  2. kernel/encoder.rs — the main Encoder struct, which provides two distinct algorithms for scalar encoding: a rotational method (via encode_scalar_log/encode_scalar_linear → encode_scalar_positional) and a transformer-style positional method.

The WAT spec in std/scalars.wat declares the three public interfaces:


(encode-log value) → Vector
(encode-linear value scale) → Vector
(encode-circular value period) → Vector

The WAT is a spec stub only (15 lines, no implementation). All real logic is in Rust.

Two Independent Encoding Systems

System A: ScalarEncoder (scalar.rs)

A standalone struct that generates two random bipolar vectors (base and orthogonal) using ChaCha8Rng seeded via SHA-256. It encodes a scalar as a rotation between these two vectors.

Vector generation (scalar.rs:79–98):


seed + name  →  SHA-256  →  first 8 bytes → u64  →  ChaCha8Rng
each dimension: rng.next_u32() & 1 == 0 ? +1.0 : -1.0

Two vectors are generated: "scalar_base" and "scalar_ortho".

System B: Encoder (encoder.rs)

The main encoder delegates scalar encoding to encode_scalar_positional, a transformer-style sinusoidal method. It does not use random base/ortho vectors — it uses deterministic sin/cos at geometrically spaced frequencies.

These two systems produce different vectors for the same input. They are separate code paths.

How Each Encoding Works

1. encode-linear — Linear Positional Encoding

In ScalarEncoder (scalar.rs:109–132)


let frac = value / scale;
let angle = frac * 2π;
result[i] = sign(base[i] * cos(angle) + ortho[i] * sin(angle))

The value is divided by scale to get a fractional position [0, 1], which is multiplied by 2π to get an angle. The output is a cosine/sine interpolation between base_vector and ortho_vector, thresholded to bipolar.

Key property: "scale determines the range: values from 0 to scale map to orthogonal vectors. Values beyond scale wrap." (scalar.rs:35)

In Encoder (encoder.rs:480–481)

encode_scalar_linear is a thin wrapper:


pub fn encode_scalar_linear(&self, value: f64, scale: f64) -> Vector {
    self.encode_scalar_positional(value, scale)
}

encode_scalar_positional (encoder.rs:591–612) uses transformer-style encoding:


for each dimension i:
    freq = 1.0 / scale^(i / dim)
    if i is even: value = sin(position * freq)
    if i is odd:  value = cos(position * freq)
    data[i] = sign(value)   // → {−1, 0, +1}

Each dimension oscillates at a different frequency, geometrically spaced from 1/scale down to 1/scale. The bipolar thresholding quantizes the continuous sin/cos wave.

2. encode-log — Logarithmic Encoding

In ScalarEncoder (scalar.rs:174–208)


pub fn encode_log(&self, value: f64) -> Vector {
    self.encode_log_base(value, 10.0)
}

pub fn encode_log_base(&self, value: f64, base: f64) -> Vector {
    if value <= 0.0 { return Vector::zeros(dim); }
    let log_value = value.log(base);
    // "10 orders of magnitude maps to one full rotation"
    self.encode_linear(log_value, 10.0)
}

Takes log-base-10 of the value, then delegates to the rotational linear encoder with a fixed scale of 10.0. The comment states: "values differing by 10^10 are orthogonal". The log step means equal ratios produce equal similarity drops.

Non-positive values return a zero vector.

In Encoder (encoder.rs:464–469)


pub fn encode_scalar_log(&self, value: f64, scale: f64) -> Vector {
    if value <= 0.0 { return Vector::zeros(self.dimensions()); }
    let log_value = value.log10();
    self.encode_scalar_positional(log_value, scale)
}

Takes log₁₀, then delegates to the transformer-style positional encoder with a caller-specified scale (default 1000.0 from the JSON marker path). Uses a different underlying algorithm than ScalarEncoder::encode_log.

3. encode-circular — Circular Encoding

In ScalarEncoder (scalar.rs:135–156)


let normalized = (value % period) / period;
let angle = normalized * 2π;
result[i] = sign(base[i] * cos(angle) + ortho[i] * sin(angle))

Nearly identical to encode_linear, but with an explicit modulo wrap: value % period ensures the endpoint wraps to the start. The doc says: "Values wrap at period, so 0 and period are identical." (scalar.rs:43)

In Encoder (encoder.rs:489–588) — Time Encoding

The Encoder does not expose a standalone encode_circular method. Instead, circular encoding appears inside encode_time, which decomposes a Unix timestamp into hour-of-day (period 24), day-of-week (period 7), and approximate month (period 12):


// encoder.rs:522–540 (inline closure)
let encode_circular = |value, period, base, ortho| -> Vector {
    let normalized = (value % period) / period;
    let angle = normalized * 2π;
    result[i] = sign(base[i] * cos(angle) + ortho[i] * sin(angle))
};

The time encoder uses deterministic role vectors from VectorManager for base/ortho (e.g., "__time_base_hour__", "__time_ortho_hour__"), not the ChaCha8Rng-based vectors from ScalarEncoder.

The four components (hour, dow, month, position) are role-bound with named vectors ("__time_role_hour__" etc.), summed as f64, then thresholded to bipolar.

WAT Spec vs Rust Implementation

WAT DeclarationRust (ScalarEncoder)Rust (Encoder)

The WAT signatures match ScalarEncoder's API more closely than Encoder's. Encoder adds an extra scale parameter to log encoding and uses a fundamentally different algorithm.

Additional Methods

encode_f64 (scalar.rs:283–296)

Returns a Vec without bipolar thresholding — preserves the continuous rotation. The doc says: "Use for f64-space comparison when extracting scalars from accumulated prototypes."

decode_log (scalar.rs:226–278)

Inverse of encode_log via grid search: coarse pass over log10 space, then refinement around the best candidate. Uses agreement count (matching non-zero signs) rather than dot product: "to avoid norm bias from zero dimensions that positional encoding produces" (scalar.rs:218).

encode_with_seed (scalar.rs:302–311)

Creates a per-name seeded encoder for encoding multiple independent scalar types (e.g., temperature vs pressure) without interference.

JSON Scalar Markers (encoder.rs:423–453)

The JSON path supports inline scalar markers:

  • {"$log": 100} → log10 encoding
  • {"$linear": 50} → linear positional encoding
  • {"$log": 100, "$scale": 5000} → custom scale parameter (default scale is 1000.0)

Key Properties

  1. Bipolar output: All encodings produce {−1, 0, +1} vectors. The zero entries occur at sin/cos zero-crossings (rotational) or exact zero values (positional).
  1. Similarity gradient: Closer values → higher cosine similarity. This is verified by tests: "sim(0,10) > sim(0,50)" (scalar.rs:331).
  1. Circular wrap: Hour 23 is more similar to hour 0 than to hour 12 (scalar.rs:350).
  1. Determinism: Same seed → same vectors. "test_deterministic" confirms ScalarEncoder::with_seed(4096, 42) is reproducible (scalar.rs:403).
  1. Log ratio preservation: Equal ratios give approximately equal similarity drops. Tests verify this with tolerance 0.1 for quantization effects (encoder.rs:948).
  1. Quantization awareness: The source explicitly acknowledges that "bipolar thresholding causes quantization" (scalar.rs:207) and that log encoding "works best when used with accumulators that average multiple samples, recovering gradient behavior" (scalar.rs:364–365).
