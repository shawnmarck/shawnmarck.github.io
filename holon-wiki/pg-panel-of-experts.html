Panel of Experts EXPLAIN

TL;DR#

The <a class="wikilink" href="T1-001-wat-language.html">wat language</a> uses a two-reviewer panel to evaluate every proposal. Brian Beckman reviews through the lens of algebraic/category-theory rigor — asking whether a proposal composes with the existing <a class="wikilink" href="pg-reckoner.html">monoid</a> and what categorical construction it represents. Rich Hickey reviews through the lens of simplicity — asking whether a proposal complects concerns and whether existing forms already suffice. Each produces an independent verdict (ACCEPTED / CONDITIONAL / REJECTED). When they disagree, a datamancer arbitrator synthesizes a final resolution, sometimes overriding both.

The Two Reviewers#

Brian Beckman — Algebraic Rigor#

Philosophy: Every language form should have a clear algebraic identity. If you can't name the categorical construction, you don't understand what you're adding.

Beckman frames proposals in category theory: monoids, coalgebras, functors, catamorphisms, universal constructions. He sees mathematical structure where Hickey sees convention.

Core questions:

  • Does this compose? — with <a class="wikilink" href="pg-bind.html">bind</a>/<a class="wikilink" href="pg-bundle.html">bundle</a>, with fold, with existing <a class="wikilink" href="T1-001-wat-language.html">Structural Forms</a>
  • What categorical construction is this? — product? coproduct? coalgebra? type class?
  • Does it introduce new algebraic structure? — if yes, is that structure justified?

Pattern of judgment:

  • Tends to accept proposals that reveal hidden algebraic structure (e.g., protocols as type classes — the "third construction" alongside struct/product and enum/coproduct, proposal 014).
  • Tends to push proposals toward core when they involve opaque types (journal operations as co-generators of a coalgebra, proposal 004).
  • Tends to prefer explicit types as a way of making structure visible and composable.

Notable positions:

  • Proposed fold as the catamorphism — the universal pattern for reducing structure (001 review, accepted as 002).
  • Framed protocols as type classes completing the categorical trinity with struct (product) and enum (coproduct) (014).
  • Argued journal operations (observe, predict, decay) are co-generators of a coalgebra and belong in Core (004 — won).
  • Wanted mandatory types on struct fields (007 — lost to Hickey).

When overridden: On struct field types (007) and defservice (04/003). In both cases, the simplicity argument prevailed: types as documentation rather than enforcement, and functions + select subsuming a dedicated service form.

Rich Hickey — Simplicity / Anti-Complecting#

Philosophy: Distinguish simple (one concept, one job, interleaving nothing) from easy (familiar, convenient, available to hand). Build simple; don't add what discipline or data restructuring already solve.

Hickey applies the "simple made easy" framework from his Clojure design work. He protects the six <a class="wikilink" href="T0-002-primitives.html">Primitives</a> from proliferation and guards against forms that bundle multiple concerns into one interface.

Core questions:

  • Does this complect? — does it weave together things that should be separate?
  • Could existing forms do this? — is a new form truly necessary, or can functions + data suffice?
  • Is this simple or easy? — convenience is not a justification if it entangles concerns.

Pattern of judgment:

  • Tends to reject new forms when discipline, data restructuring, or function composition could achieve the same result.
  • Tends to win arguments about leaving things untyped or unconstrained.
  • Tends to resist additions to the core layer, preferring stdlib or userland solutions.

Notable positions:

  • Rejected defprocessor flatly: “The function shape (state, event) → state IS the declaration” (001 — won, proposal rejected).
  • Won untyped struct fields: “Types are documentation, not enforcement” (007).
  • Won keyword-as-function access: “Keywords read like English” (007).
  • Argued fold is a caller concern, not a callee declaration (002 — partially reversed when fold was accepted as a <a class="wikilink" href="pg-vocabulary.html">control form</a>).

When overridden: On journal operations (004 — the opacity argument convinced the arbitrator they're core), nil/boolean handling (013 — datamancer chose structural absence), and channels (003 — deleted as contradicting fold-based design).

How Reviews Work#

Independent verdicts#

Each reviewer writes a separate review-beckman.md and review-hickey.md. Reviews are not rubber stamps. The two reviewers have:

  • Reversed proposals (001 — Beckman accepted, Hickey rejected; Hickey's rejection won).
  • Overruled each other (007 — Hickey won on types; 004 — Beckman won on journal placement).
  • Both been overruled by the datamancer (013, 04/003).

Each verdict is one of:

  • ACCEPTED — the proposal should proceed as written
  • CONDITIONAL — accepted with specific modifications or constraints
  • REJECTED — with reasoning and often suggestions for alternative approaches

The characteristic tension#

The Beckman-Hickey tension is productive and by design:

Dimension Beckman Hickey
Instinct What's the structure? What's the simplest thing that works?
Growth bias Add form if algebraic identity is clear Don't add if functions + data suffice
Type stance Types make structure visible and composable Types are documentation, not enforcement
Core expansion Justified by opaque types and coalgebra Resist unless necessity is proved
New forms Welcome if they complete a categorical pattern Reject if they package multiple concerns

This is not a left-right spectrum. Both reviewers agree on foundational principles (transparency to Rust, protect the algebra, the four-layer model). They disagree on how to apply those principles — particularly on whether a proposed form reveals necessary structure or introduces unnecessary ceremony.

Resolution: How Disagreements Are Settled#

The Datamancer#

When the two reviewers disagree, a datamancer arbitrator synthesizes the final RESOLUTION.md. This role is not a third voter or tiebreaker — it's a designer who looks at the problem space itself and sometimes finds that neither reviewer's position is right.

The datamancer has produced three kinds of resolution:

  1. Side with one reviewer — the simpler or more structural argument wins. Example: proposal 001, Hickey's rejection of defprocessor was upheld.

  2. Synthesize a third option — the debate reveals a path neither reviewer proposed. Example: proposal 013, the datamancer chose "structural absence" (the ? field convention, when not executing produces no value) over both Beckman's Rust-style nil and Hickey's Clojure-style truthiness.

  3. Show both are unnecessary — the simplest answer is that no new form is needed. Example: proposal 04/003, the datamancer demonstrated that <a class="wikilink" href="pg-vocabulary.html">select</a> + functions + spawn subsume both defprocess and defservice, retiring both proposed forms entirely.

Resolution outcomes#

Verdict Meaning
ACCEPTED With specific form syntax, placement, and constraints
REJECTED With open questions forwarded to future proposals
ACCEPTED with tension Accepted, but a specific disagreement is recorded and deferred
SUPERSEDED The debate itself revealed a simpler answer

Every resolution documents who was overridden and why. This creates an audit trail of design reasoning — not just what was decided, but whose argument was stronger and why.

Historical Record#

Across 18 resolved proposals (March–April 2026):

Reviewer Verdicts accepted Times overridden
Beckman fold (002), journal→core (004), protocols (014), compilation plan (005) struct types (007), defservice (04/003)
Hickey defprocessor rejected (001), untyped structs (007), keyword access (007), fold-as-caller (002 partial) journal→core (004), nil (013), channels (003)
Datamancer Structural absence (013), select+functions over defprocess/defservice (04/003) —

Both reviewers have been overturned roughly equally. The datamancer has intervened only twice, but both interventions shaped fundamental design decisions.
